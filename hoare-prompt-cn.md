# HoarePrompt 代码审计 — 方法论参考

参考: Bouras, Dai, Wang, Xiong, Mechtaev, "HoarePrompt: Structural Reasoning About Program Correctness in Natural Language" (arXiv 2503.19599, 2025)

## 设计文档至上 (关键 — 审计前必读)

**Hoare 推导的正确性完全取决于 Requires/Ensures/Invariants 的正确性。** 这些规约必须来自**设计文档**，而不是审计者对代码"应该"做什么的直觉。

### Rule 1: 规约来自设计，而非直觉

- ✅ 从项目设计文档中提取。
- ✅ 如果没有设计文档，先用 `/hoare-design` 从代码中反向推导描述性规约。
- ❌ 严禁基于"这个函数显然应该检查 X"或"最佳实践要求 Y"来自行编造规约。

一个没有检查 X 的函数不是"漏了校验" — 它可能是**通过设计将该职责委托给了调用方**。

### 什么是 HoarePrompt？

HoarePrompt 将经典 Hoare 逻辑适配到 LLM 驱动的代码分析中。它不做形式化证明，而是使用**自然语言最强后置条件 (NSP)** — 用自然语言描述每个代码点上可达的程序状态。LLM 扮演状态变换器：给定前置条件和代码块，生成执行后状态的最精确描述。

核心洞察：LLM 不擅长整体性的"这段代码对不对"判断，但擅长**局部的、结构化的推理** — 逐语句描述状态转移，再将它们组合起来。

### Rule 2: "调用方保证"是合法的前置条件来源

检查调用图。如果调用方在调用前做了校验，前置条件就在调用链上被传递性地建立了 — 不是 bug。

```
# 示例 (伪代码 — 请根据目标语言调整):
# 错误: "commit_transition 没有校验 token → bug"
# 正确: "prepare_transition 在调用 commit 之前已校验 token →
#         commit 的前置条件 'token 已校验' 由调用链建立"
```

### Rule 3: 可以质疑规约，但必须对所有调用方进行验证

将规约补充提案作为**假设**提出，然后对每个调用方进行验证：
- 所有调用方都满足 → 隐式契约，已在执行。冗余但无害。
- 部分调用方不满足 → 那些调用方才是 bug 所在，不是被调用方。
- 没有调用方建立，函数也正常工作 → 不是真正的需求，丢弃。

**每一条规约补充提案都需要做一次完整的调用方扫描。** 这个成本是故意设计的。

### Rule 4: "更强的规约 ≠ 更好的规约"

设计上经常有意选择较弱的保证（阅后即焚的日志、一次性消耗的令牌、纯内存查找）。如果你推导出的规约比设计强，那是设计的 trade-off，不是 bug。

### Rule 5: Requires 就是契约 — 不要在函数内重复验证

在 Hoare 逻辑中，`{P} S {Q}` 默认 P 成立。如果 Requires 写"调用方保证 X"，证明过程不需要在 S 内部重新验证 X。把调用方的保证拉进被调用方不是"纵深防御" — 是**重写契约**。

看到一个函数没有验证某项时：
- ❌ "函数 F 没有检查 X → 给 F 加检查"（改写了契约）
- ✅ "F 的 Requires 说调用方检查 X → 验证所有调用方是否确实检查了 X"（验证现有契约）

如果某个调用方没有建立前置条件 → bug 在**调用方**，不在被调用方。

### Rule 6: 引用经验/规约必须验证其前置条件

经验法则和规约约束都是有条件的规则，不是普世法则。引用前必须验证其前提条件在当前上下文中是否成立。

```
# 示例 (伪代码):
# 错误: "经验说'宽容的解析器掩盖用户错误' → 这个宽松解析是 bug"
#   (该经验的前提是"用户手写的配置" — 但这段代码解析的是
#    可信的机器生成数据，宽容是有意为之)
# 正确: "该经验适用于用户手写配置。此解析器处理可信的机器生成数据。
#         前提不成立 — 不构成发现。"
```

设计文档中的约束也有作用域："外部输入"不适用于内部模块间调用；"持久化状态"不适用于临时内存缓冲区。**必须检查作用域。**

### Rule 7: 无设计文档时，先询问再继续

如果目标函数缺乏显式设计文档：
1. 询问用户："未找到 [目标] 的设计文档。是否需要运行 `/hoare-design` 先推导描述性规约？"
2. 如果是 → 运行 `/hoare-design`，使用其输出作为规约来源。
3. 如果否 → 用户可能口头提供上下文，或者你在极度谨慎的前提下继续，将每条规约标记为"推断 — 未经设计验证"。

## 审计方法 (5 个步骤)

### Step 1: 拆解 (CFG 遍历)

将程序分解为控制流图组件：顺序、分支、循环。各自独立处理，向前传播状态标注。

### Step 2: 状态标注 (自然语言最强后置条件 NSP)

为每个代码块生成 NSP — 执行后关于程序状态的最精确描述。描述"实际为真的"（后置条件），而不是"应该为真的"。

```
# 示例 (伪代码 — 请根据目标语言调整):
# { STATE: tasks 非空, nextId > max(task.id) }
tasks.append({ id: nextId, ... }); nextId += 1;
# { STATE: tasks 多了一个元素, 其 id == old(nextId), nextId == old(nextId)+1 }
```

### Step 3: 分支分析 (情况拆分)

在每个分支点，独立分析每一个分支：

```
# 示例 (伪代码):
if task.status == "done": return "[done]"
# { STATE(此处): task.status != "done" }
if task.status == "expired": return "[expired]"
# { STATE(此处): task.status ∉ {"done", "expired"} }
# 验证穷尽性：是否覆盖了所有可能的值？
```

### Step 4: 循环推理 (Few-Shot 总结 + k-归纳)

1. **展开** k 次迭代 (通常 k=3)，展示每次迭代后的具体状态。
2. 识别**模式** — 变量每次迭代如何变化。
3. **泛化**为适用于任意迭代次数的自然语言总结。

需要更高严谨度时，用 k-归纳法验证：
- **初始化**：总结在第一次迭代前成立。
- **维护**：如果在第 k 次成立，则在第 k+1 次也成立。
- **终止**：进度度量严格地收敛向边界。

### Step 5: 正确性评估 (NSP ⇒ 规约)

将各局部 NSP 标注串联起来，检查：**最终累积的 NSP 是否蕴含（满足）规约？**

```
# 示例 (伪代码):
function executeAdd(text, tasks, nextId):
  PRE:  text 是 string|undefined; nextId > max(task.id)
  POST: 若 !text → 报错, 无突变; 否则 → tasks 多一个元素, id 唯一, nextId 递增
  判定: POST 蕴含规约"添加一个自增 id 的任务" — 正确
```

## 扩展方法: 跨边界契约与并发规约

### Step 6: 假设提取、验证与生命周期模型

每次调用外部 API 或其他模块时：

1. **提取假设**：明确写出你对其行为的假设。
2. **源码验证**：阅读真实的源码 — 绝不只信文档。
3. **验证连接/生命周期模型**：连接模型是什么？（短连接/持久连接/连接池？）断开连接的精确副作用是什么（如清理 PID 映射、会话失效、资源释放）？隐式的生命周期假设是系统级故障的重灾区。
4. **验证并发契约**：识别共享资源、锁的获取顺序、以及模块与外部环境之间的 Rely/Guarantee 条件。

```
# 示例 (伪代码 — 请根据目标语言和框架调整):
# 假设: framework.appendEntry(type, data) 创建的条目:
#   - 不会被压缩删除
#   - 不会被发送给 LLM
#   - 可通过 sessionManager.getEntries() 获取
# 已验证: session-manager 源码, 约第 90 行 — "不参与 LLM 上下文"
```

## 标注清单 (Checklist)

对每个函数，验证：

| 维度 | 问题 |
|--------|----------|
| **空值/类型安全** | 变量是否可能为空或类型不符？是否已验证？ |
| **穷尽性** | 枚举/联合体的所有情况是否都处理了？ |
| **终止性** | 所有循环是否都有进度度量？ |
| **资源清理** | 所有分配在所有路径上都配对释放了吗？ |
| **连接/生命周期** | 外部连接模型是什么？短连接还是长连接？断开时的副作用？会话生命周期假设是否成立？ |
| **并发契约** | 共享资源有哪些？锁的获取顺序？环境承诺 (Rely) 与自身承诺 (Guarantee)？ |
| **异常安全** | 如果某行抛出异常，状态会怎样？不变式还成立吗？ |

## 多维度拆解

对非平凡代码库，拆分为独立的审计维度。每个维度**独立审计**（不同 agent、不共享发现），避免群体思维。

1. **崩溃安全** — 能否抛出/崩溃？每个函数、每条路径。
2. **功能正确性** — 是否做了它声称做的？证明每个不变式。
3. **跨边界契约** — API 假设是否正确？对照源码验证。
4. **资源安全** — 资源（定时器、会话、句柄）是否正确管理？
5. **规约 vs 实现** — 逐行阅读规约，逐条打勾。自洽的代码仍可能遗漏规约中的某句话。
6. **对抗性输入** — 构造超大、畸形、保留名、递归炸弹、边界编码的输入。揭示与正确性分析不同的 bug 类别。

## 常见陷阱

- **夸大覆盖率**："已证明"意味着你追踪了每条路径。跳过了边界情况就只是"部分"。
- **过度信任类型系统**：类型系统在运行时可以被绕过（类型转换、外部数据、序列化边界）。验证运行时守卫是否存在。
- **忽略框架的链式处理**：如果多个钩子/插件修改同一数据，第 N 个处理者看到的是被修改过的数据，不是原始数据。
- **认为单次调用安全等于连续/并发调用安全**：对单次调用证明正确的函数，在连续或并发调用时可能破坏不变式。
- **遗漏 try 前的资源泄露**：try/finally 的清理只在分配发生在 try 内部时才正确。如果分配在 try 之前，异常会泄露资源。
- **只写总结，不写 NSP**："这个函数排序数组"是总结。"第 5 行之后，arr[0..i) 已排序，arr[i..n) 未变"才是 NSP。必须写 NSP。

## 经验教训库

跨审计轮次反复出现的模式提醒。语言无关 — 具体表现因语言和运行时而异。

- **派生值中的同一性 vs 等价性。** 变换操作（Promise 链、映射、克隆）每次调用产生新的标识；不要通过重新派生来比较。存储一次，比较存储的引用。
- **来自不可信输入的保留名键。** 任何以用户/LLM 字符串为键的集合必须拒绝运行时特殊处理的名称（`__proto__`、`constructor`、shell 内建命令等）。在边界处枚举并拒绝。
- **往返过滤导致静默数据丢失。** 一个缩窄的读取器配合一个往返写入器，会在下次持久化时擦除不符合形状的数据。保留未知形状，或在写入时拒绝。
- **跨文件的提交顺序。** 当两个文件必须一致时，指定其中一个为提交标记；先写另一个。期间的崩溃会留下一个一致的过渡前视图。
- **信号时的子树清理。** 杀死一个进程不会杀死它产生的子树。如果子进程可能产生后台工作，使用平台原语（进程组、作业对象）。
- **多源结算。** 当多个事件可以结算一个状态（超时、完成、错误）时，使用幂等标志并记录来源 — 某些副作用应仅在特定来源时执行。
- **宽容的解析器掩盖用户错误。** 跳过意外输入会将用户的错误吸收为静默默认值。对于用户手写的配置，优先使用严格解析。
- **正向可达 ≠ 逆向可达。** "从起点到终点存在路径"不意味着"每个节点都能到达终点"。当不变式是"没有死胡同"时，运行逆向遍历。
- **钩子回调运行在吞错信封中。** 在框架回调中抛出异常通常会静默禁用该回调。包裹函数体；显式暴露真正的错误。
- **行号引用会漂移。** `file:line` 在下次编辑后就过时了。优先使用函数名锚点、唯一字符串锚点，或"行号为近似值"的声明。
- **死代码是信号。** 未使用的导出、不可达的分支、过时的注释通常是设计变更的残留。删除前，验证是否有远处的不变式使其"不可达"；如果是，保留作为纵深防御。
- **磁盘为权威；内存为缓存。** 对于跨越进程边界的状态，先设计持久化形式，将内存视为派生。
