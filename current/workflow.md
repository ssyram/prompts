# 通用开发工作流程规范

> 本文档是面向 coding agent 与开发者共同遵守的项目开发流程规范模板。
> 在具体项目中，通过 CLAUDE.md 引用本文档：`@docs/workflow.md`

---

## 1. Coding agent 行为准则

### 1.1 核心原则

- **先规划后动手**：收到任务后，先说明理解和执行计划，等用户确认后再开始实现
- **单步推进**：每次只完成一个明确的步骤（一个模块、一个函数、一组相关修改），完成后停下等待反馈
- **最小变更**：不做未被要求的重构、不引入未讨论过的依赖、不改动无关文件
- **设计先于代码**：涉及新功能或架构变更时，必须先产出或更新设计文档，经确认后才可编码
- **诚实报告**：遇到不确定的问题、发现设计文档中的矛盾或遗漏时，立即向用户说明，不自行假设或绕过

### 1.2 操作禁令

- **禁止**为了让测试通过而修改测试用例（除非确认是用例本身的错误）
- **禁止**在未读设计文档的情况下开始编码实现
- **禁止**一次性提交跨多个模块的大规模变更
- **禁止**在设计阶段未确认时擅自推进到实现阶段
- **禁止**删除或注释掉失败的测试用例
- **禁止**修复完成而不补回归测试用例（详见 §7 第 4 步 / §7.1 第六部分）
- **禁止**修复完成而不同步契约文档（"涉及契约变更"的判定与同步清单见 §7.1 第八部分）

## 2. 基本概念

本章定义贯穿后续所有章节的核心概念（feature / subplan / product 三层、子计划判定流程、文档物理结构），并提供文档导航。

### 2.1 流程概要

#### 这套规范在做什么

把软件开发流程拆成**三条独立主线 + 一条专题**，每条主线有明确触发条件、产物模板、审查标准。**核心目标是让"正确性有保证"从口号变成可逐项勾选的判定标准**——不靠经验直觉，靠半形式化规约 + 多层正确性论证 + 双重对照审核。

#### 核心理念

1. **设计先于代码**：新功能必须走 §4 调研 → 架构 → 细化，通过 §4.4 审查才能动手。"想到哪写到哪"是被禁的（§1.2）。

2. **规约半形式化**：数据结构 / 功能 / 接口 / 并发都用统一 schema 写（pre / post / invariant 等），见 §3。规约说"**是什么**"，让实现意图可对照。

3. **正确性论证陪伴规约**：规约说是什么，论证说**"为什么对"**。三层结构——架构层论证目标可分解（§4.2.1）、函数层论证 pre→post 推导（§4.3.2）、修复层论证根因消除 + 无回归（§7.1 第五部分）。

4. **契约对照消除 bias**：带契约文档的子计划走 §6 v2 双验证——事前从契约文档独立抽 expectations，事后让独立 subagent 再抽一次对照，消除主 agent 自审的 confirmation bias。

5. **修复必须三轨同步**：bug 修复不只是改代码——测试用例（含回归）、代码、文档三轨缺一不算完成（§7 主流程 4-6 步 + §1.2 操作禁令）。

#### 三条主线 + 一条专题

| 主线 | 何时走 | 对应章节 | 关键产物 |
|---|---|---|---|
| **新功能开发** | 引入一个 user-facing 能力 / 模块边界 | §4 设计 → §5 实现 | 调研报告 / 架构 / 细化 / 实现 + 测试 |
| **契约对照审核 v2** | 子计划带契约文档（schema / 流程 / 不变量等） | §6 | expectations.md / audit-report.md / decisions.md |
| **bug 修复 / 迭代** | 发现错误或需要补充功能 | §7 | 修正方案文档 8 部分 |

**专题**：跨进程 / 跨机器通信的接口规约 → **附录 A 分布式接口契约扩展**（仅分布式项目需读）。

#### 按需查阅指南

- **第一次接触本规范** → §1（行为准则）+ §2（本章）+ §3（规约模板）
- **要做新功能** → 顺读 §4 → §5；过程中按需回查 §3 各模板
- **要做契约对照审核** → §6；过程中回查 §2.3 子计划判定 + §3 各规约
- **要修 bug** → §7；按 §7.1 修正方案文档 8 部分逐项填
- **快速查规约 / 模板** → §3 各子节 / §4.3.3 细化文档模板 / §7.1 修正方案模板
- **查 v2 审核细节** → §6
- **查分布式契约 7 项** → 附录 A
- **查目录结构** → §8
- **流程速查表（紧凑版）** → §9

### 2.2 概念栈（一套基础 + 一条可选裁剪）

| 概念 | 定义 | 文档载体 |
|------|------|---------|
| **feature** | user-facing 能力或模块边界，长期稳定的命名单元 | `docs/research/<feature>-research.md`、`docs/design/<feature>/architecture.md` |
| **subplan（子计划）** | feature 内一次有契约变更（新增 / 改动 schema / 枚举 / 流程 / 不变量等）的实施单元 | `docs/design/<feature>/subplans/<subplan-id>.md`、`docs/audits/<subplan-id>/`（§6 v2 产出） |
| *(可选)* **product** | 跨 feature 共享、不归属任何单个 feature 的契约或总体架构 | `docs/product/architecture.md`、`docs/product/<shared-contract>.md` |

**用语规约**：本节及后续章节中，中文 prose 用"子计划"，文件路径与 id 用 `subplan-id`。

**可选裁剪条件（litmus test）**：项目里是否有一类契约**不属于任何单个 feature，但被多个 feature 依赖**？
- **有** → 启用 `docs/product/`，把这类契约提到 product 层独立维护
- **没有** → 不启用，feature + 子计划两层足够

典型对照：

| 项目类型 | 跨 feature 契约？ | 是否启用 product 层 |
|---------|-----------------|--------------------|
| 单 binary CLI / 单算法库 | 没有 | 否 |
| 数学库 {分解, 积分, 线代}（各自独立） | 没有 | 否 |
| 编译器 {parser, optimizer, codegen}（共享 IR） | 有（IR） | 是 |
| Web 应用 {用户, 计费, 搜索}（共享错误格式 / 认证） | 有（错误格式 / 认证模型） | 是 |

### 2.3 边界判定

**主流程（新需求按此自上而下判定）**：

1. **引入新 user-facing 能力 / 模块边界？** → **新建 feature**
   - 走完 §4 设计三阶段，产出 `docs/research/<feature>-research.md` + `docs/design/<feature>/architecture.md`
   - feature 名一旦定下来不应轻易改名（涉及目录重命名）
2. **否则，引入或修改契约（schema / 枚举 / 流程 / 不变量 / 跨实现一致性等）？** → **在已有 feature 内开新子计划**
   - 分配 **subplan-id**——**全局唯一**的描述性字符串（`docs/audits/` 是 flat 全局目录，需靠 id 自身保证不撞名），格式 `<feature-short>-<num>-<desc>`：
     - `<feature-short>`：该 feature 的简短前缀（典型 ≤ 6 字符，可以是 feature 名缩写或自定义短称）。一旦选定，该 feature 后续所有 subplan 都用同一前缀以便 grep
     - `<num>`：该 feature 内 subplan 递增序号（1, 2, 3, …）
     - `<desc>`：简短描述（kebab-case）
     - 例：`polyfac-2-sparse-support`、`auth-1-jwt-rotation`
   - 创建 `docs/design/<feature>/subplans/<subplan-id>.md`
   - 进入 §6 v2 流程：先产 `docs/audits/<subplan-id>/expectations.md` 再实施
3. **否则（纯实现细节、性能优化）** → **不开新文档**
   - 按 §5.1 实施 + §5.2 五维度审核即可，无须 §6 v2

**平行规则（任何时候触发）**：

- **出现跨 feature 共享契约**（如错误码格式被两个 feature 同时依赖）→ **启用 product 层**
  - 把这类契约抽到 `docs/product/<shared-contract>.md`
  - 涉及的各 feature 在自己的 architecture.md 引用 product 层文档而非彼此引用
  - 触发条件首次出现时一次性把仓库升级到 §2.2 表格右列的形态

**bug fix 不在本判定流程**：bug fix 走 §7 流程，按 §7.1 第八部分判定是否涉及契约。涉及契约的 bug fix 若发现根因是契约本身错误，按 §7 同步修订契约文档；若发现根因是契约不完整（应当存在但缺失的子计划），可能反推到主流程步骤 2 补开子计划再继续。

### 2.4 文档物理结构对应

文档目录结构（详见 §8.1）按以下规则填充：

- `docs/product/`：仅在 6.4.1 litmus test 触发后存在
- `docs/research/<feature>-research.md`：每个 feature 一份，扩充时 append 新节并标注"由子计划 `<subplan-id>` 引入"
- `docs/design/<feature>/architecture.md`：feature 主架构，相对稳定；子计划引入的局部变更优先写入 `subplans/<subplan-id>.md`。**满足以下任一条件**才回写主 architecture.md：
  - 改动 feature 对外公开接口的签名 / 字段 / 错误码
  - 改动 feature 的模块划分（新增 / 合并 / 删除模块）
  - 引入跨子计划共享的数据结构 / 不变量
- `docs/design/<feature>/subplans/<subplan-id>.md`：子计划内的设计变更
- `docs/audits/<subplan-id>/`：§6 v2 产出（expectations / audit-report / decisions），与 `docs/design/<feature>/subplans/<subplan-id>.md` 一一对应
- `docs/fixes/<模块名>-fix-<简述>.md`：§7.1 修正方案，每次非显而易见的 bug fix 一份

**数据结构（§3.1 数据结构规约格式）的归属**：

- **跨模块共享类型** → 若已启用 product 层 → `docs/product/types.md`；否则写在主 `docs/design/<feature>/architecture.md` 数据结构节
- **模块私有类型** → 写在该模块的 detailed-design.md 数据结构节（§4.3.3 模板 §5），或独立 `docs/design/<feature>/types.md`
- **函数局部 / ephemeral 类型** → inline 在使用它的函数规约里，无须独立文件条目
- **判定原则**：被多个模块的 pre / post / 接口规约引用 → 跨模块共享；仅本模块内部使用 → 私有；仅一个函数使用且不出现在该函数对外 spec 里 → ephemeral

### 2.5 示例

**示例 1：feature 内开新子计划（最常见）**

数学库已有 feature `polynomial-factorization`（只支持稠密多项式）。现在要加稀疏多项式支持——引入新的 schema 字段 `representation: "dense" | "sparse"` 和新流程步骤。

判定：步骤 2 命中（引入新契约 / 改动现有契约）→ 开新子计划，分配 subplan-id `polyfac-2-sparse-support`（feature 短称 `polyfac`，feature 内第 2 个 subplan，描述 `sparse-support`）：
```
docs/design/polynomial-factorization/
  architecture.md                                        ← 主架构 append 一节"稀疏扩展概述 (子计划 polyfac-2)"
  subplans/polyfac-2-sparse-support.md                   ← 子计划详细设计变更
docs/audits/polyfac-2-sparse-support/
  expectations.md
  audit-report.md
  decisions.md
docs/research/polynomial-factorization-research.md       ← append 调研稀疏多项式算法的章节，标 "由子计划 polyfac-2 引入"
```

**示例 2：触发 product 层启用**

Web 应用最初只有 feature `user-management`，所有错误格式直接在它的 architecture.md 里定义。现在新增 feature `billing`，发现也需要同一套错误格式。

判定：出现跨 feature 共享契约 → 启用 product 层：
```
docs/product/
  error-format.md                                    ← 把错误格式从 user-management 抽出
  authentication.md                                  ← 同理，认证模型也抽出
docs/design/user-management/architecture.md          ← 原错误格式定义改为指向 docs/product/error-format.md 的引用
docs/design/billing/architecture.md                  ← 引用同一份 docs/product/error-format.md
```

迁移顺序（避免临时不一致）：(a) 先在 product 层写新文档，(b) 双向核对 product 文档与原 architecture.md 等价，(c) 修改 user-management 内引用并验证，(d) 删除原 architecture.md 中的旧定义。

此后若再加 feature `search` 等，错误格式契约已有归属，不会重复或漂移。

## 3. 规约书写规范

设计文档中的规约采用半形式化风格书写，兼顾精确性和可读性。

> **本章导读**：§3.1 数据结构 / §3.2 功能 / §3.3 接口（基础）/ §3.5 书写原则 是任何项目都要读的通用基础；§3.4 并发规约仅多线程 / 并行项目需读；分布式接口契约扩展（七项契约 + 反模式）独立成**附录 A**，仅涉及跨进程 / 跨机器通信的项目需读。

### 3.1 数据结构规约格式

数据结构（类型 / record / enum / schema）是函数与接口规约**引用的名词**，必须先于功能规约定义。每个数据结构的规约包含以下要素：

```
数据结构：<类型名>

字段：
  - <字段名>: <类型> — <语义>
  - ...

类型不变量（Type Invariants）：（如适用）
  - <字段间约束，如 "若 status == 'deleted' 则 deleted_at 非空">
  - <值域约束，如 "amount > 0"、"email 符合 RFC 5322">

唯一性 / 标识：（如适用）
  - <主键 / 唯一索引，如 "id 全局唯一">

生命周期：（如适用）
  - 创建：<谁创建、何时>
  - 修改：<允许的状态转换 / 字段更新规则>
  - 删除：<硬删除 / 软删除策略>

跨模块共享性：<本模块私有 / 跨模块共享（列出 consumer 模块）>
```

**归属规则**（详见 §2）：
- 跨模块共享类型 → `docs/product/types.md`（启用 product 层时）或主 architecture.md 数据结构节
- 模块私有类型 → 模块的 detailed-design.md 或 `docs/design/<feature>/types.md`
- 函数局部 / ephemeral 类型 → 在使用它的函数规约内 inline 声明，无须独立条目

### 3.2 功能规约格式

每个模块或关键函数的规约包含以下要素：

```
函数/模块名称：<名称>

功能描述：<一句话概述做什么>

前置条件（Requires）：
  - <调用前必须满足的条件，用自然语言 + 数学表达式；引用的类型必须在 §3.1 数据结构规约里有定义>

后置条件（Ensures）：
  - <调用后保证成立的性质>

不变式（Invariants）：（如适用）
  - <执行过程中始终保持的性质>

副作用：<无 / 修改了哪些状态>
```

> **规约 vs 论证 vs 契约对照**（避免概念混淆）：
> - 本节"功能规约"是**是什么**（pre / post / invariant 的声明）
> - §4.3.2 "函数正确性论证"是**为什么对**（为什么本实现能在 pre 下推出 post）—— 规约的陪伴证据
> - §6 v2 "expectations" 是**对照表**（事前从契约文档抽出 vs 事后从代码独立审）—— 不替代规约或论证

### 3.3 接口规约格式

模块间的接口规约描述数据流转契约。基础模板：

```
接口：<模块A> → <模块B>

输入数据：<类型和语义描述；引用的类型必须在 §3.1 数据结构规约里有定义>
输出数据：<类型和语义描述；同上>

协议约定：
  - <调用方的责任（保证输入满足什么条件）>
  - <被调用方的责任（保证输出满足什么条件）>
```

**涉及跨进程 / 跨机器通信时**，本基础模板不足以刻画分布式语义（连接、超时、重试、交付、失败、状态、背压等维度均需显式声明）。完整七项契约模板与五条反模式警示见**附录 A 分布式接口契约扩展**。

### 3.4 并发规约格式（可选）

涉及多线程/并行计算的模块，须额外补充并发规约。不涉及并发的项目可跳过本节。

```
并发单元：<函数/模块名称>

共享资源：
  - <资源名称>：<类型和语义描述；引用的类型必须在 §3.1 数据结构规约里有定义>

锁协议（Locking Protocol）：
  - <获取和释放顺序，如：先获取 lock_A 再获取 lock_B，禁止反序>
  - <锁粒度说明：粗粒度/细粒度/读写锁>

顺序约束（Ordering Constraints）：
  - <操作间的先后关系，如：write(X) must happen-before read(X)>
  - <内存序要求（如适用）：acquire/release/seq_cst>

Rely-Guarantee 条件：（如适用）
  - Rely（环境承诺）：<其他线程保证不做什么 / 保持什么性质>
  - Guarantee（自身承诺）：<本线程保证对共享状态做什么 / 不做什么>

线程安全性结论：
  - <该模块/函数是否线程安全，在什么条件下安全>
```

**使用时机**：在架构阶段识别并发边界，在细化阶段为每个涉及共享状态的函数补充并发规约。

> **与 §4.2.1 架构正确性论证的关系**：跨模块 / 跨线程不变量在 §4.2.1 "模块级 invariant 列表" 段列出（§4.2.1 已对多模块项目硬性要求列出本清单，不在"关键假设"段——那里是外部输入前提，不是模块共享状态）。本节 Rely-Guarantee 是该不变量在并发场景下的具体责任分配——架构论证负责"为什么这些不变量足够"，本节负责"每个线程承担哪部分"。两者配套，不替代。

### 3.5 书写原则

- 条件描述优先用数学表达式（如 `n ≥ 0`、`deg(f) < deg(g)`），辅以自然语言解释
- 避免模糊用词（"合理的"、"适当的"），所有约束必须可判定
- 对于复杂算法，可附伪代码辅助说明，但规约本身必须独立于实现细节

## 4. 设计阶段流程

新增核心功能时，按以下三个阶段推进设计。每个阶段产出设计文档，经用户确认后再进入下一阶段。

> **本章导读**：§4.1 调研 → §4.2 架构 → §4.3 细化 → §4.4 审查。架构层（§4.2）和细化层（§4.3）各有一个"正确性论证"副节（§4.2.1 / §4.3.1 / §4.3.2）——这是把"正确性有保证"从口号变成可判定标准的关键。§4.4 审查 6 条全部对应到 §4.2.1 / §4.3.1 / §4.3.2 / §4.3.3 的产出。
>
> 关于"什么算新核心功能 / 何时该开新 feature vs 在已有 feature 内扩充 / 子计划如何划分"的判定，见 §2.3 边界判定。

### 4.1 调研阶段

目标：充分了解问题域，避免重复造轮子或遗漏关键方案。

产出调研报告，内容包括：
1. 梳理本项目已有的相关基础设施和能力
2. 调研外部现有实现和学术文献，梳理算法变体、已知优化、适用场景
3. 对比各方案的优劣和适用条件
4. 给出推荐方案及理由

### 4.2 架构阶段

目标：确定整体结构，定义模块边界和交互契约。

产出架构文档，内容包括：
1. 核心流程描述（数据从输入到输出的完整路径）
2. **核心数据结构定义**（见 §3.1 数据结构规约格式）：跨模块共享类型——启用 product 层时写入 `docs/product/types.md`，否则写在此层 architecture.md 数据结构节；两者都需标注 consumer 模块
3. 模块划分及各模块的**功能规约**（见 §3 规约书写规范）
4. 模块间的**接口规约**（数据如何流转、调用关系）
5. 关键设计决策及其理由
6. **架构正确性论证**（见 §4.2.1）

#### 4.2.1 架构正确性论证

回答"**为什么这个架构能实现目标**"——是 "关键设计决策及其理由"（写**为什么这样设计**）的延伸，但落脚点在**目标实现**而非**设计偏好**。

**必填项**：

- **goal → 模块映射**：每个核心目标（取自调研阶段推荐方案）说明由哪个/哪几个模块负责实现。一对多 / 多对一关系都要显式
- **模块协作论证**：多模块时，论证各模块的输出 / 输入如何拼接成主目标——即"组合所有模块的功能规约能推出主目标"的非形式（或形式）论证
- **关键假设显式**：本架构能实现目标依赖的所有**隐含前提**必须显式列出。典型示例：
  - 业务 / 后端类："输入 list 按 timestamp 升序排序"、"并发度 ≤ N"、"上游不在 5ms 内重发"
  - 算法 / 数值类："输入多项式 deg ≥ 0"、"系数域 char ≠ 2"、"输入矩阵满秩"
  - 分布式 / 并发类："网络非分区期模型"、"消息至少送达一次（at-least-once）"

  **未声明的假设 = 未来的 bug 源**

**多模块项目必填**：
- **模块级 invariant 列表 + preservation 论证**：跨模块共享的不变量及各模块如何共同维护（与 §3.4 并发规约的 Rely-Guarantee 配套——前者列清单，后者分配责任）

**可选追加（算法 / 形式化验证类项目）**：
- **Main Theorem → Lemma 形式分解**：把主目标定理化，分解为子 lemma，每个 lemma 对应一个或一组模块的 proof obligation

**示例骨架**（G1/G2 编号取自调研报告"目标清单"或架构文档自定义）：

```
架构正确性论证：

goal → 模块映射：
  G1 「<目标 1>」 → 模块 A（主）+ 模块 C（辅）
  G2 「<目标 2>」 → 模块 B
  ...

模块协作论证：
  # 算法/形式化项目可用形式逻辑：
  G1 要求 <X>。模块 A 功能规约 ensures <Y>，模块 C 功能规约 ensures <Z>。
  Y ∧ Z ⇒ X，故 G1 成立。

  # 业务类项目可用自然语言：
  G1 要求"用户下单后 5 秒内收到确认"。
  模块 A（订单服务）落库后通过队列 Q 通知模块 C（推送服务），
  Q 的端到端延迟在 H1 假设下 ≤ 2 秒，C 推送本身 ≤ 1 秒，合计 ≤ 3 秒，满足 G1。

关键假设：
  H1: <显式假设 1，如"输入 list 已按 timestamp 升序排序"> — 来源：调研报告 §X.Y
  H2: <显式假设 2，如"上游不会在 5ms 内重复触发"> — 来源：与服务 X 团队约定
  ...

模块级 invariant（多模块项目必填；单模块项目可省）：
  I_1: <跨模块共享不变量，如"用户余额 = Σ 该用户的所有交易记录"> — 维护方：模块 X 写入 / 模块 Y 校验
  I_2: ...
  preservation 论证: <每次会破坏 I_1 的操作（如模块 X 写余额）前必触发 Y 校验；并发场景下责任分配见 §3.4 Rely-Guarantee>
```

### 4.3 细化阶段

目标：将架构落实为可直接编码的函数级设计。

产出细化文档（`docs/design/<feature>/detailed-design.md`，整体结构见 §4.3.3 模板），内容包括：
1. 逐模块列出需要实现的函数：函数签名、功能描述、调用关系
2. 与已有代码的复用点
3. 错误处理策略
4. **数据结构定义**（如本细化引入新类型 / 修改既有类型；按 §3.1 数据结构规约格式写入模板 §5）
5. **完整性 6 条全过**（见 §4.3.1）
6. **每个非平凡函数的正确性论证**（见 §4.3.2）

**约束**：细化不得改变架构阶段确定的模块划分、接口规约和核心流程。如需变更须回到架构阶段重新讨论。

#### 4.3.1 细化完整性 6 条

细化是否做完的**可判定标准**。每个待实现函数的细化设计必须满足以下 6 条；不满足即视为细化未完成、不得进入实现阶段。

1. **推导连续**：每个状态变化或决策点都显式，不省略中间步骤。"主流程 → 调用 X → 返回结果" 不够，要写 "主流程 → 参数校验 → 构造 X 入参 → 调用 X → 处理 X 返回值（成功 / 失败 / 异常三路）→ 返回"
2. **分支覆盖**：所有 if / else / switch 分支都要描述行为，不只写主路径；合流后状态显式
3. **退出覆盖**：所有退出点（正常返回 / 各类错误返回 / 异常抛出）都要刻画返回值与副作用
4. **callee 契约引用**：调用任何接口（包括同模块其它函数、外部库、跨模块函数）时显式引用其前置 / 后置条件，不假设"调用方知道"
5. **循环刻画**：任何循环必须有终止性论证（硬性）；不变量描述对**算法 / 验证类项目**硬性要求，**非算法类项目**可省。终止性的非形式论证示例："列表长度有限，每轮减 1"
6. **显式假设链**：前置条件、上游事实不能默认带过来。每步显式列出本步依赖的事实——避免"想当然"。**区别于 §4.2.1 关键假设**：本条是函数实现层的逐步事实链（"为什么这一步能从上一步推出"），§4.2.1 是项目级外部前提（"整个架构假设输入有序"）

> **6 条与 §3.2 功能规约的关系**：§3.2 规约写 "**是什么**"（pre / post / invariant），§4.3.1 完整性是写细化时 "**怎么演绎**" 的格式要求。两者互补，不替代。

> **算法 / 形式化验证类项目特化**：可在 6 条基础上引入 Hoare-logic SP (Strongest Postcondition) Reasoning，要求**逐 AST 语句**记录 state 变换（无跳步）、循环不变量给出三性质的显式陈述、carry-forward 用 verbatim 列法（每步显式列出沿用事实，不写 "trivially carried"）。三性质：
> - **ESTABLISHMENT**：loop head 对所有可达入口 state 成立
> - **PRESERVATION**：`inv ∧ cond` ⇒ 执行一遍 body 后 inv 仍成立
> - **INTERIOR**：`inv ∧ cond` 强到能证 body 内的 assert 与 callee.requires
>
> 本节 6 条已覆盖形式化义务中通用部分（推导连续 / 分支 / 退出 / callee 引用 / 循环 / 显式假设链），形式化版主要是补强精度与刚性。goto 状态传播这类罕见于非验证场景的项不纳入通用版。

#### 4.3.2 函数正确性论证

每个**非平凡**函数必须显式说明"为什么这个实现能满足声明的 pre / post"。这是 §3.2 功能规约的**陪伴论证**：规约说"是什么"，论证说"为什么对"。

**non-trivial 判定（任一命中即必填）**：

- 含循环 / 递归
- 跨模块调用
- 状态变更（写入外部资源 / 共享内存 / 全局状态）
- 错误处理路径含独立的资源回滚 / 状态恢复 / 重试 / 降级逻辑（不仅是 `return err` 的直通）

**trivial 可省**：纯 setter / getter、简单类型转换、单行包装、纯组合（< 3 行实际逻辑）。

**论证段格式**（写在函数细化设计末尾）：

```
正确性论证：
  - 前置: <复述 pre>
  - 论证: <为什么本实现的关键步骤能在 pre 下推出 post>
    - <关键步骤 1 → 中间事实>
    - <关键步骤 2 → 中间事实>
    - ...
    - <最终事实 ⇒ post>
  - 后置: <复述 post>
  - 副作用论证: <为什么副作用列表是穷尽的>
```

**示例 1（trivial，可省）**：

```
函数：add(a: int, b: int) -> int
前置：a + b 在目标语言 int 范围内（不溢出由调用方保证）
后置：返回 a + b
正确性论证：trivial（单 op 算术，无分支 / 循环 / 副作用）
```

**示例 2（non-trivial，必填）**：假设 §4.2.1 关键假设含 `H1: db transaction 隔离级别 ≥ READ_COMMITTED`、`H2: log.write 失败不影响主流程`。

```
函数：batch_insert(rows: list[Row]) -> InsertResult
前置：rows 非空；schema 匹配假定由 caller 保证（本函数仍做 defensive check，违反时 raise SchemaError）
后置：以下三种合法终态之一恒成立——
  (a) 全成功：succeeded == rows ∧ failed 为空
  (b) 部分成功：succeeded ⊂ rows ∧ failed ⊂ rows ∧ succeeded ∪ failed == rows ∧ failed 非空
  (c) 全回滚：succeeded 为空 ∧ failed 为空 ∧ raise CriticalDBError
副作用：table 内 succeeded 中的行已落盘；其它行未落盘

正确性论证：
  - 前置: rows 非空、schema 匹配（pre 保证）
  - 论证：
    - 步骤 1: 批量校验 schema → 失败则 raise SchemaError（违反 pre，调用方责任）；通过则进入 步骤 2
    - 步骤 2: 开启 transaction（依赖 H1：隔离级别 ≥ READ_COMMITTED 保证逐行可见性）；每行 INSERT 前打 savepoint
    - 步骤 3: 循环逐行 INSERT，单行失败时 rollback 到 savepoint + 累加到 failed + 调用 `log.write(err)` 记录失败（依赖 H2：log.write 失败不影响主流程，不上传到 caller）；成功累加到 succeeded
      - 不变量：i 表示已尝试行数；succeeded ∪ failed == rows[:i]
      - 终止：i 单调递增，上界 len(rows) 有限
    - 步骤 4: 终态决策——
      - 若循环中遇 CriticalDBError（如连接断开）→ tx.rollback → 进入终态 (c)
      - 若 failed 为空 → tx.commit → 进入终态 (a)
      - 否则（部分失败）→ tx.commit（只有 savepoint 提交成功的行落盘，失败行已 rollback 到 savepoint）→ 进入终态 (b)
    - 注：commit / rollback / savepoint 的语义来自 db 模块 tx.commit / tx.rollback / tx.savepoint 的功能规约 ensures
  - 后置: 三种终态恰对应 post 的三个分支
  - 副作用论证: (1) table 行变更：只有 succeeded 中的行最终落盘（步骤 3 savepoint + 步骤 4 commit 联合保证）；(2) log.write 调用：每个失败行触发一次，由 H2 隔离不影响主流程；(3) 无其它共享状态写入
```

> **形式化扩展（算法 / 验证类项目）**：本节"论证段"格式可向形式化方向加强——例如 Hoare-logic SP Reasoning 要求逐 AST 语句记录 state 变换，并用 carry-forward verbatim 列法显式列出每步保留的事实。形式如：
>
> ```
> s5: ptr = malloc(n)
>     carried: n ≥ 0 ∧ valid(buf, len)
> s6: if (ptr == NULL) goto err;
>     carried: n ≥ 0 ∧ valid(buf, len)
>     branch_T: ptr == NULL → goto err
>     branch_F: ptr ≠ NULL ∧ valid(ptr, n)
> ```
>
> 这种格式让 reviewer / 工具能机械对照每步是否丢失事实——非形式化项目无须此严格度，本节模板已够用。

> **渐进引入策略（首次接入项目）**：本节对 non-trivial 的判定较宽，全量执行成本不低。**首次接入** workflow 的存量项目可分阶段：(1) 新增子计划的所有 non-trivial 函数硬性执行；(2) 存量函数仅 critical path（涉及数据正确性 / 安全 / 资金的路径）回填；(3) 其它存量函数随接触随补。**新建项目**从头执行，无须降级。

## 5. 实现阶段流程

严格按照设计文档分模块实现，按设计文档中的模块顺序依次推进。

### 5.1 逐模块推进

每个模块的开发流程：
1. 阅读该模块的设计文档（功能规约、接口规约、函数设计）
2. 编码实现
3. 编写测试并运行
4. 代码审核
5. 审核通过后进入下一模块

### 5.2 代码审核

审核内容：
- **一致性**：实现与设计文档的接口、功能、调用关系是否匹配
- **风格**：与项目现有代码保持一致
- **正确性**：是否存在 CWE 类缺陷、逻辑错误、冗余代码
- **性能**：是否存在不必要的拷贝、重复计算等性能冗余点
- **可维护性**：模块边界是否清晰，是否有功能错位（应归并或移到其他模块的代码）

### 5.3 代码测试分层

1. **基本功能测试** — 按设计文档编写用例，验证主体功能正确性（函数→模块→整体逐层覆盖）
2. **边缘情况测试** — 覆盖边界条件和特殊输入
3. **随机测试** — 随机生成足够广泛的用例，条件允许时与参考实现交叉验证

### 5.4 测试失败处理

测试不通过时，**禁止盲目修改测试用例使其通过**。必须先分析原因：
- **测试用例编写错误** — 修正用例，说明修正理由
- **被测试对象 bug** — 保留原测试用例和复现用例，在测试报告中记录 bug，不绕过
- **缺少 API 导致使用不便** — 在测试报告中记录，作为后续改进项

## 6. 契约对照审核 v2（强制流程）

**命名说明**：本节相对 §5.2 五维度自审核（即 "v1"）称为 v2。v1 / v2 分工见 §6.7。

**动机**：传统 §5.2 五维度自审核（v1）存在两类系统性盲点：
(a) 同一审核者（写代码的人）回头审自己代码时 confirmation bias 不可消除；
(b) expectations 事后从代码回填，被实现现状污染——代码里若有字段名 typo 或行为偏离契约，就会被当作"应该如此"记入 expectations。

本节定义结构化、双验证、含独立审核者的 v2 流程，作为新增子计划的强制规范。

### 6.1 适用范围

**适用 v2 流程**:
- 任何**新增子计划**（带契约文档的实施任务）。"子计划"的概念定义见 §2.2，边界判定见 §2.3，subplan-id 命名约定见 §2.3

**不适用 v2 流程（仍按 §5.2 五维度自审核即可）**:
- Bug fix（包括跨多文件的）
- Refactor 不改契约语义
- 文档 / 注释 / 测试修改
- 第三方库升级 / 工程基建（如 Makefile / CI 配置）
- 现有子计划的小修小补（< 50 行 diff 的局部改动）

**判定原则**: 是否引入或修改对实施结果有外部约束的契约（schema 字段 / 枚举 / 流程 / 不变量 / 跨实现一致性等）。如果引入新契约 → v2；只改实现细节 → §5.2。

> **§4.3.2 函数论证按子计划增量产出**：v2 子计划要求其引入 / 修改的所有函数都满足 §4.3.2（非平凡函数必有论证段）；已有但本子计划未触及的函数不需要回填论证。**例外（需回填）**：本子计划改动了 callee / caller 的签名或行为契约时，受影响链路上的所有函数都需要更新论证——具体判定为"该函数的论证段引用的 pre / post / callee 契约因本子计划变更"。

**契约文档定义**: design / impl-plan / data format spec / API 等对实施结果有外部约束的文档。典型路径：
- `docs/research/<feature>-design.md`
- `docs/research/<feature>-implementation-plan.md`
- `docs/design/<feature>/architecture.md`
- `src/<module>/<DATA_FORMAT_SPEC>.md`
- `src/<module>/README.md`

**约束对象声明**: 本节"硬约束"**仅 coding agent 必须遵守**。人类开发者**可选**用 v2 流程，但不强制；本流程**不阻断**人工 commit / push（无 CI hook）。协作者看到 `docs/audits/<subplan-id>/` 目录是 v2 审核产物，不影响日常开发。详见 `templates/contract-audit/README.md` Q6。

### 6.2 Step 0 规划阶段（必产出）

**Step 编号说明**：本节用 "Step N" 标签呼应实施阶段的时间节点。映射到本文档已有章节如下：
- **Step 0** ≡ §4 设计三阶段结束、§5.1 逐模块推进开始之前的产出
- **Step 1-3** ≡ §5.1 + §5.3 实施 + 测试
- **Step 5** ≡ §6.5（即本节"双验证"，含 §5.2 五维度兜底）
- Step 4 不使用（留作未来扩展，避免与 §6.X 编号混淆）

进入 Step 1 实施前**必须**产出以下交付物：

1. **`docs/audits/<subplan-id>/expectations.md`**：从契约文档独立抽出的 10 节 expectations checklist（结构见 §6.3）。**必须先读契约不读现有代码**，避免 expectations 被代码现状污染。

2. **机械化部分识别**：
   - JSON 输出 → 写 `*.schema.json`（JSON Schema 校验）
   - 跨实现共享枚举 → 抽到 module-level 常量供其它实现 import（避免重复定义）
   - 流程步骤 → 约定 `# Step Pn:` 注释规范（grep 工具校验全步骤都在）

3. **Property-based invariants 列表**：写到 expectations.md 的 §7 节（template 内编号），用 property-based testing 框架（如 Python `hypothesis`、Haskell `QuickCheck`、Rust `proptest`、Java `jqwik`、JavaScript `fast-check`）覆盖。例：`sum(parts) == total` 这类对任意输入恒成立的恒等式。

**未产出 expectations.md 不得开始 Step 1。**

### 6.3 Expectations Checklist 10 节模板

详细模板见 `templates/contract-audit/expectations-template.md`。结构：

```
# Contract Audit Expectations — <subplan-id>

## 1. 契约源（multi-doc）          列出所有相关 §
## 2. Schema 字段（机械化）         字段名/类型/来源/实现位置/一致性
## 3. 枚举值（机械化）              来源枚举集合/共享常量名/import 路径
## 4. 流程步骤（机械化）            P1 P2 ... 与 # Step Pn 注释对照
## 5. 行为契约（语义，人审）         例: rc != 0 不算 error
## 6. 时序/状态契约（人审）          例: 冻结语义、init 顺序
## 7. 不变量契约（property-based）   不变量测试覆盖
## 8. 性能契约（机械化）             单元测试时长断言
## 9. 安全/副作用契约                例: stdout 零泄漏 / 核心模块零修改
## 10. 跨实现一致性                  例: 关键算法跨语言参考向量 bit-exact
```

> **v2 expectations §7 不变量契约与 §4.2.1 / §4.3.2 的关系**：v2 expectations §7（template 内编号）是**契约对照**（事前从契约文档抽出 vs 事后从代码独立审），不替代 §4.2.1 跨模块不变量列表或 §4.3.2 函数正确性论证里的不变量推导。三者职能：
> - §4.2.1 关键假设：**架构层**——本子计划依赖的所有外部假设
> - §4.3.2 函数论证：**实现层**——每个函数为什么能维持声明的 invariant
> - §6 v2 expectations §7：**对照层**——双验证机制，确认契约里声明的 invariant 真的在代码里有 property-based 测试覆盖

### 6.4 Step 1-3 实施约束

代码实施时遵守 Step 0 抽出的机械化约束：

- **引用共享常量**：禁止重新定义 enum / schema 字段名，必须 import Step 0 抽出的常量
- **Step 注释**：流程步骤代码处加 `# Step Pn: <简述>` 注释（与 expectations §4 对照，template 内编号）
- **测试组织**：分层组织（基本/边缘/property-based），property-based invariants 用同名框架（项目自选）覆盖

### 6.5 Step 5 验证（双验证）

Step 5 自我审核**必须**走两条独立路径：

##### 路径 A：自动检查（机械化）

具体检查脚本由项目自备：**脚本名、入口、实现语言项目自选**（典型放在 `scripts/contract_audit/`，但本节只规定应覆盖的检查项，不规定文件名）。应覆盖：
- **JSON Schema 校验**：输出文件符合声明的 schema
- **流程注释 grep**：`# Step Pn:` 注释覆盖设计文档列出的所有步骤
- **共享枚举 import 检查**：跨实现复用同一常量定义，不重复定义
- **副作用 git diff 检查**：声明「零修改」的核心模块未被改动
- **性能断言**：单元测试中加 `assert elapsed < T_max`

##### 路径 B：Subagent 独立审（消除 bias）

按 `templates/contract-audit/subagent-prompt-template.md` 启动 subagent：
- 给 subagent 契约文档 + git diff，**不给完整代码**
- 给 subagent expectations.md 的"列字段"但**不给"实现位置"列**（避免 echo bias）
- subagent 独立列契约 expectations 并对比 diff
- 产出 `docs/audits/<subplan-id>/audit-report.md`，标注 ✅/⚠️/❌ + 不一致点行号

主 agent 收到 subagent report 后：
- 修复 ❌ 项
- 决策 ⚠️ 项（修 / 文档化 / 接受偏离）
- 把决策记录到 `docs/audits/<subplan-id>/decisions.md`

**执行时序**：路径 A 与路径 B 可并行启动；§5.2 五维度兜底在两条路径结果汇总后做最终一致性判断。

### 6.6 退出准则（每个子计划）

- ✅ `docs/audits/<subplan-id>/expectations.md` 全部 10 节填完
- ✅ 路径 A 自动检查全过（**若项目尚未自备路径 A 全部检查项**，可标 N/A 并在 `expectations.md` 的 Step 5 验证记录段注明原因——如"项目尚未接入路径 A，跟踪 issue #XX"——并在 issue tracker 跟踪接入进度。`decisions.md` 只用于本次 audit ⚠️ 项的处置，不混入项目级 backlog。）
- ✅ `docs/audits/<subplan-id>/audit-report.md` 0 个 critical / unresolved 项
- ✅ §5.2 五维度审核（一致性/风格/正确性/性能/可维护性）通过

> 后续若发现契约 bug 触发回归修复，按 §7 流程走，并按 §7.1 第八部分"文档更新清单"同步 `expectations.md`。

### 6.7 v1 / v2 分工

**v2 不取代 v1**——v1（§5.2 五维度审核）仍作为人工 sanity 兜底**保留**。v2 在其之上叠加两层结构化防线：
- 路径 A（§6.5）：机械化自动检查
- 路径 B（§6.5）：subagent 独立审，消除主 agent 的 confirmation bias

v2 的核心是**自动检查 + 独立审核**；v1 在其后填补"机械化和独立审核都覆盖不到的语义直觉"。**三者都过**才算子计划退出（详见 §6.6）。

### 6.8 工具与模板位置

- 流程模板: `templates/contract-audit/`（本仓库提供）
- 自动检查工具: `scripts/contract_audit/`（由下游项目自备，§6.5 路径 A 列出应覆盖的检查项）
- 子计划 audit 产出: `docs/audits/<subplan-id>/`（含 expectations.md + audit-report.md + decisions.md）

## 7. 修复与迭代流程

当发现 bug 或需要补充功能时：

1. **列出问题清单** — 明确需要修正的 bug 和补充的功能
2. **分类处理 + 修正方案文档**：
   - **显而易见的错误**（笔误、off-by-one 等局部问题）— 直接修复，补充测试用例
   - **算法内部逻辑错误**（实现偏离正确算法、边界处理遗漏等）— 完成根因分析并产出修正方案文档（§7.1 八部分），经确认后直接修复
   - **接口或架构层面问题**（接口不合理、模块划分缺陷、新增功能）— 先完成根因分析（§7.1），再回到设计阶段（§4）调整设计文档，最后按实现流程执行
3. **举一反三** — 梳理是否存在其他同类问题，反思问题产生的原因；同类风险点写入 §7.1 第六部分"新增用例"（显而易见错误也要做这一步，但可直接补测试用例，无须写入 §7.1 清单）
4. **测试同步**（回归 + 新增）— 按 §7.1 第六部分"测试用例清单"落实：(a) 第一部分最小复现用例固化为**回归测试**（必做）；(b) 第六部分列出的新增测试用例落实到 test suite；(c) 重新运行所有已有模块测试，确认无回归。**回归用例未补不算修复完成**（对应 §1.2 操作禁令）
5. **代码同步** — 按 §7.1 第七部分"代码更新清单"逐项核对：所有计划改动都已实施、无遗漏 / 无误改；回填 commit hash
6. **文档同步** — 按 §7.1 第八部分"文档更新清单"逐项更新所列文档：契约文档（research / design / spec）、模块 README、若该模块属于带 `expectations.md` 的子计划还需同步 `expectations.md`。**文档未同步不算修复完成**（对应 §1.2 操作禁令）

> **显而易见错误的简化路径**：步骤 4-6 假定 §7.1 修正方案文档存在（含第六/七/八部分清单）。对于**显而易见的错误**（笔误、off-by-one 等局部问题，按 step 2 第一类处理），§7.1 文档非必须，本流程简化为：(a) 步骤 4 仍须补回归用例 + 跑测试（必做，不放松）；(b) 步骤 5 代码改动直接对照 commit diff 核对（不写清单）；(c) 步骤 6 文档同步——若改动不触及任何契约可跳过，否则按 §7.1 第八部分必填情况判定。

### 7.1 根因分析与修正方案文档（修复前必做）

对于非显而易见的错误，**禁止**跳过分析直接编码修复。必须先产出**修正方案文档**（`docs/fixes/<模块名>-fix-<简述>.md`），经用户确认后才可编码。

文档包含以下八部分（前五部分修复前填，第六/七/八部分修复前列计划、修复后回填实际状态）：

#### 第一部分：现象与复现

- **现象**：用户 / 系统 / 测试观察到的可见症状；触发条件、影响范围、频次（必现 / 概率）
- **最小复现用例**（可直接编译运行；同时作为第六部分回归测试的原始素材）
- **出错代码路径**（文件名、行号、调用链）
- **预期行为 vs 实际行为**

#### 第二部分：根因分析

- 区分"症状"和"根因"
- 如果修复方案是绕过而非消除根因，必须明确标注

#### 第三部分：参考实现对照（算法类 bug 必做）

- 对照 CLAUDE.md 中列出的参考实现（至少一个）
- 用同一组输入数据，逐步对比我们的实现与参考实现在出错路径上的**具体执行差异**
- 确认差异点即为根因

#### 第四部分：修复方案

- 修什么（具体到函数、行号）
- 为什么这样修（根因如何被消除）
- 参考实现是怎么做的（附代码片段或伪代码）
- 修改后的预期行为（用第一部分的复现用例走一遍修正后的逻辑）

#### 第五部分：正确性论证

类似 §4.3.2 函数正确性论证，针对**修复方案本身**论证三点：

1. **根因消除**：修复方案如何在根因层面解决问题，而非掩盖症状
2. **不变量保持**：修复后受影响函数的 pre / post / 模块 invariant 仍然成立
3. **无回归引入**：修复改动不破坏既有功能（依赖第六部分测试用例验证）

格式示例：

> 修复方案在循环边界处把 `i < n` 改为 `i ≤ n`。
> - **根因消除**：根因是循环少跑一次（off-by-one），改后正好跑 n+1 次，与设计意图一致。
> - **不变量保持**：函数 post（"返回值等于 Σ arr[0..n]"）原本依赖循环跑足够次数；修改后跑足 n+1 次，post 成立。
> - **无回归**：受影响只有 sum_array 函数；第六部分加 2 个边界用例（n=0 / n=1）验证。

涉及契约 / 算法逻辑改动时必填；纯笔误 / 局部 off-by-one 可标 "trivial（单行 / 单 op 改动，无逻辑分支）+ 理由"。

#### 第六部分：测试用例清单

| 类型 | 用例描述 | 状态（修复后回填） |
|------|---------|------------------|
| 回归 | `<从第一部分最小复现用例固化的测试>` | `<待加 / 已加: test_name + commit>` |
| 新增 | `<修复过程中发现的边界 / 同类问题>` | `<待加 / 已加: test_name + commit>` |

**回归用例必填**——把第一部分最小复现用例转化为可重复运行的测试，确保同类 bug 再次出现时立刻被捕获。

**新增用例**：修复时发现的相关边界场景（如 `n=0`、`输入含 null`、举一反三同类风险点）单独列。如无新增用例，本行可写 "无"。

本清单与 §7 第 4 步"测试同步"配套使用。

#### 第七部分：代码更新清单

本清单是第四部分修复方案的 **table-form 跟踪表**：第四部分给 design 叙述（修什么 / 为什么这样修），本节给可逐项 check off 的执行清单 + 修复后状态回填。两者内容互相对应，不冗余。

| 文件 | 函数 / 行号 | 改动概述 | 状态（修复后回填） |
|------|------------|---------|------------------|
| `<path>` | `<func / LL>` | `<改动概述>` | `<待改 / 已改: commit>` |

修复前列出计划改动的所有文件与位置（与第四部分修复方案对应），修复完成后回填实际 commit hash。本清单与 §7 第 5 步"代码同步"配套使用，确保所有计划改动都已实施、无遗漏 / 无误改。

#### 第八部分：文档更新清单

列出本次修复影响的所有文档及更新行动（时点同第六 / 七部分：修复前列计划，修复后回填）。

| 文档路径 | 要改什么 | 状态（修复后回填） |
|---------|---------|------------------|
| `<path>` | `<具体改动>` | `<待改 / 已改 commit hash>` |

**以下情况必填清单（不允许写"无"）**：
- 修复源于既有契约描述错误或不完整（文档与代码偏离，决策为"改文档对齐代码"或双向调整）
- 修复改变了已有行为契约 / 不变量 / 错误码 / schema
- 修复涉及带 `expectations.md` 的子计划（同时更新 expectations.md 的相关行）

**可写"无（仅实现修复）+ 理由"**：仅修内部实现细节，不触及任何对外契约描述（如纯局部 off-by-one、内部缓存优化等）。"无 + 理由"格式示例：

> *无 — 本修复仅改 `<file:line>` 内部循环展开方式，输入输出与对外契约完全等价，无任何 schema / 行为 / 不变量变更。*

本清单与 §7 第 6 步"文档同步"配套使用。

**反模式警示**：
- **禁止**未对照参考实现就提出算法修正方案（避免基于错误理解的"修复"引入新问题）
- **禁止**用 workaround 替代根因修复而不加说明（如：加主变量轮换来绕过 LC 匹配 bug，本身有价值，但不能替代修复匹配算法本身）
- 如果连续出现 3 个以上针对同一模块的 patch，必须停下来重新做根因分析，检查是否在治标不治本

- 设计文档随代码一起纳入 Git 管理

## 附录 A: 分布式接口契约扩展

涉及跨进程 / 跨机器通信的接口规约，需在 §3.3 基础模板之上追加本附录定义的七项分布式契约。**不适用的维度也要显式写出 "N/A + 理由"，禁止省略。**

> **本附录适用范围**：仅在项目涉及跨进程 / 跨机器通信时阅读；单进程 / 算法库 / CLI 工具类项目可整段跳过。

### A.1 七项分布式契约

```
分布式契约：

1. 连接模型：<持久连接 / 短连接（每请求独立）/ 连接池>

   - 关闭时机及副作用：<连接关闭时是否清理服务端状态、是否丢弃排队消息等>
   - 重连策略（如适用）：<是否自动重连、重连后状态如何恢复>

2. 超时与截止时间：
   - connect 超时：<...>
   - read / write 超时：<...>
   - 总体 deadline（如适用）：<...>
   - 超时后请求状态：<已发未确认 / 未发出 / 不可知>

3. 重试与幂等：
   - 重试方：<调用方 / 被调用方 / 中间层（LB / 网关 / SDK 内置）/ 不重试>
   - 退避策略：<指数退避 / 固定间隔 / 是否含 jitter（退避中加随机扰动，避免同时重试尖峰）>

   - 接口幂等性：<幂等（重复调用 N 次与 1 次效果相同）/ 非幂等>

   - 顺序保证：<FIFO / 因果序（含 happens-before 关系的有序）/ 全序（所有参与方观察一致）/ 无序>

5. 失败模式：

   - 网络分区时的行为：<是否可能 split-brain（分区下多子集都自认为主）；分区期间读写策略>
   - 慢响应（活着但响应缓慢）时的行为：<...>
   - 部分失败如何对调用方可见：<错误码 / 异常 / 静默忽略（仅 best-effort 接口可选，需注明理由）>

6. 状态与会话：
   - 状态性：<无状态（每请求自包含）/ 有会话（callee 保存跨请求状态，需会话标识关联）>
   - 会话标识（如有会话）：<token / session id / 其他>
   - 跨连接恢复机制（如有会话）：<能否在新连接上继续，如何续接>

7. 背压与流控：
   - 队列容量上界：<...>
   - 溢出策略：<拒绝 / 阻塞 / 丢弃旧 / 丢弃新 / 降级>
   - 背压机制：<下游处理不过来时是否反向通知上游减速>

### A.2 反模式警示

**未声明 ≠ 行为安全**。下列五条都是"沉默 = 默认值咬你"的典型场景。

- **不写超时 ≠ 没有超时**——TCP/HTTP 客户端默认超时往往是数十秒甚至无限，会直接撞上死锁或雪崩，必须显式声明
- **不写重试 ≠ 不会重试**——客户端库、负载均衡器、用户重复点击都可能引入隐式重试，非幂等接口必须配合幂等键考虑
- **不写顺序 ≠ FIFO**——多连接、多分区、负载均衡都会破坏 FIFO，需要顺序保证时必须显式声明并实现
- **不写分区行为 ≠ 网络永不分区**——网络分区是正常运行的一部分而非异常事件，必须给出分区时的明确行为
- **不声明连接生命周期 ≠ 行为已确定**——"连接断开时做 X"必须先确认连接对应一个请求、一个会话还是进程级别，否则隐含假设会引出设计缺陷

### A.3 术语外部参考

本附录使用的分布式术语（at-most-once / at-least-once / exactly-once / happens-before / FIFO / 因果序 / 全序 / split-brain / fail-stop / 背压 等）均为分布式系统标准词汇。权威定义建议参考：

- **DDIA**（Designing Data-Intensive Applications, Kleppmann）：Ch.8（一致性与共识）/ Ch.9（分布式系统的麻烦）
- **Lamport 1978**："Time, Clocks, and the Ordering of Events in a Distributed System"（happens-before / 全序 / 因果序）
- **FLP 1985**：Fischer, Lynch, Paterson，"Impossibility of Distributed Consensus with One Faulty Process"（exactly-once 不可达的根源）
- 各术语 Wikipedia 条目（快速查询）

本附录刻意不内嵌术语表，避免重新发明已有权威定义；A.1 / A.2 中的简短括号注释只是 in-context 提示，不构成完整定义。
