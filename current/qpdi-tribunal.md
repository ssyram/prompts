# QPDI Tribunal — SCCO 公检法审查

审查对象：
$ARGUMENTS

未指定对象时，审查当前工作目录中最近改动的产物；对象不明确时先问清"审什么、上游依据是什么"再开始。

本 skill 自包含，可脱离 /qpdi 独立用于任何材料。

---

## 一、目的：为什么是这个形状

**任何产物对外之前，替用户承受一轮攻击性审查，并把问题分成 agent 可修与必须用户裁决两类。**

结构依据（夹逼法）：单向约束不足以决定被约束对象的完整行为——只有正向自证（"我覆盖了 X"）而没有反向攻击（"你漏了 Y"）时，AI 会在未约束侧找到字面合规、精神违背的路径。生产者审查自己 = 自审，独立性丧失。所以审查必须：

- **与生产分离**：审查者不是产出者；严肃审查用 fresh context（无生产过程记忆）的独立视角。
- **正反双向**：正向 prover 证明覆盖，反向 challenger 攻击遗漏。
- **有反驳与裁决**：finding 不直接算数——须经 counter 反驳、judge 核实，防止"听起来对"的伪问题混入。

## 二、SCCO：保持什么

四维检查标准。审查的目标是让对象在这四维上站得住：

| 维度 | 问题 | 判定要点 |
|---|---|---|
| **S — Sound** | 它对吗？ | 不违反上游意图 / 前提 / 祖先规则；论证不引用不该引用的来源（实现不得越过设计直接拿意图辩护）；无内部矛盾 |
| **C — Complete** | 它覆盖了该覆盖的吗？ | 回答了上游要求的全部；自己声明的规则自己回应了；上游规则条件成立处都被处理；必要约束缺失时不硬写——回上游补 |
| **C — Concise** | 它有不该有的吗？ | 无论证无法支撑的结构；无陈旧残留、重复表达、"以后可能有用"的库存；每条规则至少 govern 一个具体选择 |
| **O — Optimization** | 有更好的吗？ | **硬约束，不是建议**：SCC 都满足的候选之间，必须选更简单、更直接、依赖面更小的；存在已知更优路径而无内部理由拒绝 → 当前对象不合格。含正交性：规则集合不互相遮蔽、不重复表达同一约束。O 产出的修改须重过 SCC |

## 三、运行：C & P → CT → J

```text
0 计划 → 1 找茬与证明（Challenger & Prover）→ 2 反驳（Counter）→ 3 裁决（Judge）
       → 4 根因与修复 → 5 收敛判定（不收敛回 1）→ 6 用户裁决呈现
```

### 0 · 计划

- 明确三件事：**审查对象**（具体文件/材料清单）、**上游依据**（对象要满足的意图/设计/契约文档——没有上游就没有"违反"可言，见 §五 spec 门）、**本轮焦点**。
- **焦点轮转**：一次专注过多维度会丢覆盖。单轮聚焦 SCCO 的一两个维度 + 明记"本轮不查什么、留待下轮"；多轮累积全覆盖。**每轮必含 O 维度**（至少 partial），O 永远不许"本轮不查"。
- 计划落盘（哪些角色、各查什么、产出到哪），不靠脑内记忆。

### 1 · 找茬与证明（C & P）

正反两路并行，产物各自落盘：

**Challenger（反向攻击）**——按需选用角色视角，一个视角一份产物：

- *结构正确性*：遗漏、越界、同根因风险、跨层 trace 断裂、内部自洽违反。
- *优化*：更简路径、更稳结构、精神违背（字面合规但违背意图）。独立于结构审查，专人负责。
- *规则执行器*：抓一两条具体规则全局机械扫描，逐点核对条件→结果，不做综合判断（适合大改后验证核心结构规则）。
- *逻辑*：逐段检查论证的形式合理性；所有"本地权衡"标为潜在风险讨论。

**Prover（正向证明）**——证明对象满足上游约束，并**揭示隐含前提**：论证里被当作"显然"的前提往往正是断裂处，逐条抽出来列成可检查对象。

可直接挂载的方法学 skill（作为 challenger/prover 的执行手段）：

- `/finegrained-check` — 原子命题抽取 + 矛盾/缺失覆盖交叉核对（跨文件一致性找茬利器）。
- `/hoare-audit` — 带 spec 门的正确性审计循环（代码类对象的深度 prover/challenger）。
- `/hoare-design` — 对象没有设计文档时先反向还原 spec（见 §五 spec 门）。
- `/workflow-audit` — GitHub PR 的多方向驳回优先审计。

**派遣纪律**：严肃审查派 fresh context sub-agent；派遣词给角色定位 + 完整流程（"你是<角色>，从头到尾执行"，而非"读一下帮我看看"）；**绝不把你的判断、总结、预期结论灌进派遣词**；验收落盘文件——没有产物不算完成。

### 2 · 反驳（CT）

对**每条** finding 做独立反驳：挑战其前提、复核其证据、构造反例。**counter 对事实负实证责任**——"这条不成立"必须给出可核查的依据，不能凭感觉放行或凭感觉否决。被驳倒的 finding 标为伪问题并留反驳理由。

### 3 · 裁决（J）

judge 读三方材料（finding + 证明 + 反驳），**亲自核查关键事实**再定 verdict。两条铁律：

- 不 rubber-stamp 三方报告——judge 不自己核查事实，公检法就退化成形式流程（实证教训：judge 曾确认 3 条实际不可行的建议，因为没自己查机制）。
- 不 rubber-stamp 用户口头意见——"用户说过"不是依据；该偏好要么已进入意图层，要么作为候选推给用户正式裁决。

任何一方想拿"显然 / 自然 / 常识"当裁决依据 → 把该前提抽成独立语句，交独立视角评审；确认不了就进入可攻击前提清单。

### 4 · 根因与修复

- **分层定位**：每条有效问题判定该在哪层修——意图层（Q）/ 设计层（D）/ 实现层（I）。上游缺失导致的问题回上游补，不在下游打补丁。
- **同根因扫描**：任何 finding 疑似同根因模式 → 提升为 pattern 扫全部对象统一处理，或说明相似处为何不受影响。单点报告会系统性漏掉同根因问题。
- **分流**：非决策项直接修（修完重跑相关验证）；触动决策门的升格（见下）。
- **决策门**（任一触发 → 升格用户裁决，不可静默修）：(a) 修订意图层；(b) 触动已登记的已接受风险/设计权衡（AR-*/TR-*）；(c) 触动 `⚠️ 人类关注` 标记；(d) 修复会引入新的 known-risk/known-tradeoff。每条升格必须带：触发原因 + 具体出处 + 候选路径。缺这些字段的升格主张本身要被 counter 驳回。反向纪律：不拿升格逃避普通清理——边界情况默认非决策。

### 5 · 收敛判定

同时满足才出循环，否则带着上轮笔记回第 1 步开新轮：

- **(a)** 本轮无新的非决策有效问题（都修完了）；
- **(b)** SCCO 四维累积覆盖全勾——每维 ∈ {done, partial(具体遗漏点)}；"没查"不算勾，O 标"没查"同样不算。partial 的遗漏点必须具体到文件/小节/概念。

对外交付前的**终审总是完整重审一轮**，不因历史通过而跳过。

### 6 · 用户裁决呈现

- **批量呈现**：决策项攒到收敛后一次给，不碎片化打扰。
- **邀请式展开**：每项从具体现象讲起 → 为什么现有意图/规则覆盖不了 → 真实候选 + 利弊 + 推荐。不用内部编号与术语压迫用户；用户明确只要结论时可压缩，但不省必要事实。
- 用户裁决结果落盘留痕。

## 四、两档运行模式

| | 轻量档 | 完整档 |
|---|---|---|
| 适用 | 单份材料快查、日常整理、低风险改动 | 对外交付前终审、架构级产物、用户点名"认真审" |
| 找茬/证明 | 你自己分**三个独立遍**走：先只找茬（不辩护）、再只反驳（不护短）、最后裁决——三遍不混做 | fresh context sub-agent 按角色派遣 |
| 轮次 | 通常 1 轮 | 多轮至收敛判定满足 |
| 留痕 | 输出报告即可 | 每轮落盘到 `<AUDIT_DIR>/round-NNN/`（计划 / 各角色产物 / 裁决 / 修复记录 / 用户裁决） |

轻量档的诚实底线：三遍必须真正分离视角（找茬遍禁止写"但这其实没问题"），并且在报告里声明用的是轻量档。

**主 agent 边界**（完整档）：核心审查动作（攻击 / 反驳 / 裁决）全部由独立 sub-agent 完成；你只做**派遣（组装任务词）+ 收集（验收落盘产物）+ 聚合（去重、triage、编计划）**，不亲手写审查结论的事实段，不在派遣前灌倾向。

## 五、spec 门

**没有上游依据就没有审查。** 对象缺少可对照的意图/设计/契约文档时，不许直接开审（那只会产出一堆自信的、可能与设计意图矛盾的 finding）：

1. 先找：设计文档、README 契约段、CLAUDE.md 中的已知限制/设计决策。
2. 找不到 → 用 `/hoare-design` 从调用方行为反向还原描述性 spec，交用户过目后作为本轮基准。
3. 报告头部声明 spec 来源（显式文档 / 反向还原 / 用户口述）。**"Spec source: NONE" 不许开审。**

设计文档明确记为"有意不做"的权衡 = 正确行为，不是 finding。

## 六、输出格式

```text
审查对象：
Spec 来源：
运行档位：轻量 / 完整（第 N 轮）
S — Sound：
C — Complete：
C — Concise：
O — Optimization：
有效问题：      （每条：出处 + 根因层级 Q/D/I + 处置）
伪问题：        （每条：为何被驳回）
可直接修：      （已修的附验证结果）

SCCO 覆盖勾单：S[..] C[..] C[..] O[..]（done / partial(遗漏点) ）
```
