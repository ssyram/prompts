# 论文阅读的问题意识驱动展开

> 派生自 `/principle-derivation` (v1)，针对**已发表技术论文的阅读理解**这一具体场景特化。
>
> 与 v1 的关系：沿用 v1 的"意识 → 原则 → 推导"骨架；本命令是 v1 在论文阅读场景下的 canonical 实例化。
>
> 与 v2 (`/principle-derivation-v2`) 的关系：v2 是关于"意识展开"的抽象邀请框架；本命令是它在论文阅读这一类**实践性主体**上的具体特化形态。实践性主体的展开有有限自由度（被论文文本和实验数据约束），所以可以给出更具体的 canonical 形态，而不止是邀请。

---

# 一、适用范围

- 主体：**已发表的成熟技术论文/系统文档**（不是思辨/哲学/设计草案）
- 目标：**完整理解对方在说什么 + 把核心 Idea 内化为自身可复用的认知**
- 不适用：审稿（priority 是找问题）、辩驳（priority 是反对）、纯思辨（没有实验和文本约束）

---

# 二、核心姿态

**理解优先于评估**。先吃透对方的问题、Idea、展开、自我评估——再考虑批判性拷问。批判前置只会让阅读在自己的偏见里和论文擦肩而过。

**不安感是状态，不是工具**。读者在没拼齐三元组之前会自然不安，应该帮他拼齐，而不是在他还没拼齐时再加新拷问。拷问是自洽之后的延伸阶段，不是阅读阶段的输入。

**有源 ≥ 直接 ≥ 延伸**：
- **有源性（best）**：claim 能直接溯源到论文原文（abstract / §intro / 作者明文的 challenge / 实验数据）
- **直接性（acceptable）**：claim 不在原文，但能从原文洞察直接且较为唯一地推出
- **延伸（必须分开）**：claim 是读者的元方法归纳、跨领域类比、迁移性 hypothesis——可以有，但必须放进独立的"延伸思考"框，不能混入"论文核心理解"

---

# 三、输出形态：三元组 + 组合展开

## 3.1 三元组

| 维度 | 内容 | 锚点要求 |
|---|---|---|
| **问题** | 论文要解决什么 + 来自哪里 + 为什么是问题 | 直接溯源 §intro |
| **Insight / Idea** | 引发解法的核心创意（往往不是工程清单，是"作者发现了 X"这种洞察） | 直接溯源 §intro 的 challenge / observation 段 |
| **展开** | Idea 如何被具体方案落实（4+7 之类）；每个组件参与机制讲清楚 | 直接溯源 §method |
| **回答程度** | 实证证据（实验数字、对照、acknowledged limitations） | 直接溯源 §eval |

## 3.2 问题的三问（展开问题维度）

- **是什么**：论文具体技术问题
- **来自哪里**：从什么现象、什么先前工作的局限引发
- **为什么是问题**：为什么这个现象/局限不能被既有工具解决

## 3.3 组合展开（关键）：一元 → 二元 → 三元

如果 Insight 拆出 N 个独立要点（D1, D2, ..., D_N），不要一下子把全部组件抛给读者。**展示组合过程**：

- **一元解**：只满足 D1 时，方案长什么样、缺什么
- **二元解**：加上 D2，方案怎么扩张、新增什么组件、它和 D1 的组件怎么配合
- **三元解**：加 D3，方案完整化、整个设计的 4+7（或对应数字）的全貌出现

**关键**：每一步要讲清楚**新组件如何参与、和已有组件如何配合**——不能只列清单。配套优化（如 planner、view refinement、n-sampling）要明确说它们具体解决哪一层问题，不能笼统列出。

读者读完一元→二元→三元的展开应该能感觉到"设计是被原则一层一层逼出来的"，不是"作者拍脑袋列了一份清单"。

## 3.4 回答程度

只引用论文自己给出的：
- 实验数据（具体数字、benchmark 规模）
- 对照基线（与什么比，差距是多少）
- 论文自承的局限（§threats to validity / §limitations）

**不引入读者的批判性评估**——那是延伸阶段的事。

---

# 四、边界严格：作者 claim vs 读者延伸

写完三元组后，可以追加一个**独立**的「延伸思考」段，包含读者自己的：

- 元方法归纳（如"pipeline > agent on structured tasks"——这是读者从实验观察归纳出的 hypothesis，不是论文的 claim）
- 跨领域迁移性假设
- 反向证伪思考
- 作者没明说但可能存在的局限
- 与同类工作的更深对比

**延伸思考**段必须明确标注每条延伸的性质——它是 hypothesis 还是 conjecture，统计基础是否足够，是否需要更多领域验证才能成 claim。

**这一段可以有，但必须和三元组严格分开**。混合两者会让读者困惑哪些是论文说的、哪些是分析者说的。

---

# 五、应用步骤

1. **读论文 §intro 的 challenge / problem statement 段**——抓 Insight 的有源溯源
2. **读 §method 的 architecture / framework 段**——抓 Idea 如何展开成具体组件
3. **读 §eval / §experiments**——抓回答程度
4. **读 §limitations / §future work**——抓作者自承的边界
5. **构造三元组**——按上述 canonical 形态
6. **构造组合展开**——一元→二元→三元，配套组件归位
7. **追加延伸思考**——把读者自己的归纳放在独立段
8. **自检**（见 §七）

---

# 六、反模式

- **延伸冒充原文**：把读者的归纳混在"论文核心理解"里。检验：每一句能在 §X 找到锚点吗？不能就标为延伸
- **跳过 Insight 直接列工程**：列出 4+7 但没说背后的 Idea。读者记得住数字、记不住内核，无法内化
- **组合不展示**：直接给最终 4+7 的全貌，不展示一元→二元→三元的逐步组合。读者无法获得"为什么是这些"的认知
- **配套组件笼统化**：只说"还有 planner、view refinement、n-sampling"但不说它们各自参与什么、解决哪层问题。读者只能死记
- **过早拷问**：三元组还没拼齐就引入规模/对照/迁移性评估。这会打断理解、把读者推回不安状态
- **匀质化叙述**：每个组件同等篇幅展开，Insight 没有突出的核心地位

# 七、失败补救

| 用户反馈 | 诊断 | 补救动作 |
|---|---|---|
| "看不到 Idea，只看到工程清单" | Insight 被漏掉或被埋在细节里 | 回 §intro 找作者明文的 challenge / observation / insight 段，重新提取 |
| "我不知道这个组件怎么参与的" | 组合展开缺自洽细节 | 回到一元→二元→三元，把每个组件的参与机制说清楚 |
| "这是你的延伸还是论文的话" | 边界不清晰 | 补加溯源标注；或迁移到"延伸思考"段 |
| "组件清单读完了但没记住" | 没做组合展开，跳过了"为什么是这些"的逻辑 | 回到 D 列表，重新做一元→二元→三元 |

---

# 八、自检

写完后倒着检查：

- 三元组每一项都有 §N 锚点吗？
- Insight 拆出的 D 数量 ≤ 4 吗？多于 4 颗粒度过细应合并
- 一元 → 二元 → 三元 的展开有没有跳步？每步加的组件都解释了"如何参与"吗？
- 配套组件（planner、refinement、sampling 等）都说清楚是解决哪层具体问题的吗？
- 延伸思考是否和原文三元组严格分开？延伸的 hypothesis 性质是否标注？
- 整段读完，读者是否**能在自己脑中复现**"从 Insight 推出 4+7"这条路径？如果不能，组合展开还不到位

# 九、和 v1 / v2 的协作关系

| 任务类型 | 用 |
|---|---|
| 已发表论文的完整理解 | **本命令** |
| 已有设计的事后原则梳理（audit） | v1 |
| 一般主题的意识展开（思辨/设计/非论文） | v2 |
| 自己的设计要立结构 | v1 → 然后用 v2 的姿态打磨 |

本命令是 v1 + v2 在「实践性主体 = 论文阅读」这一具体场景上的特化。它接受了 v2 的姿态（理解优先、自洽为目标、不主动制造不安），但提供了 v1 风格的具体形态（三元组 + 组合展开）。**实践性主体之所以能给出 canonical 形态，是因为论文文本和实验数据本身就提供了硬约束**——这把"展开有无穷可能"的开放性收敛掉了。
