# charter-craft — 制宪与改宪方法学

> 用于项目根本约束 / 长期承诺文档（principles.md / 项目宪法）的制定与修订。

---

## §0 适用范围与本 skill 的性质

### §0.1 适用范围

**v1 面向**：行动性 / 项目级宪法 —— 即"为解决某个具体问题而立项"的项目根本约束。

例：软件项目的 `principles.md`、研究项目的根本约束文档、长期承诺类文档。

**根本机制**（问题驱动 + 意图捕获 + 防腐蚀）**理论上**适用更广（个人价值 / 团队规章 / 长期制度），但 v1 不 enumerate 这些领域——经验积累后再扩展。

### §0.2 本 skill 的性质

**本 skill 不是"宪法之宪法"。** 它是 AI 制宪时的工作经验集合 + 少量行为硬约束。

- **硬约束**仅在 §1 + §2（约束 AI 行为，不约束宪法内容形式）
- **实践推荐**在 §3 起后所有章节（建议性，可按场景调整）

### §0.3 关于所有"原则"的本质

**本 skill 提的所有"原则"都是理想 / 北极星，不是机械可执行的 binary check。**

- 正交性是理想 — 现实条款经常 80-90% 独立，少量交叉
- 完备性是理想 — 没有宪法能枚举所有情况
- 奥卡姆是理想 — "不必要"看判断
- 抽象层级是理想 — 边界永远模糊
- 派生而非穷尽是理想 — 完全派生几乎不可能
- 篇幅基线是经验 — 超也可以但要能解释

**反模式（执行本 skill 时的反模式）**：

- 把理想当 binary check → "这条不正交？拒绝！"
- 把流程当机械执行 → "起源 5 问没全答 → 不能继续"
- 把篇幅当上限 → "201 行了 → 必须剃到 200 以下"

应该问"它偏离理想多远？合理吗？有更好的吗？"

**承认不完美 + 承认权衡 = 制宪诚实**。

## §1 元问题意识：宪法 = 防腐蚀机制

### §1.1 宪法解决什么根本问题

**宪法 = 对"承诺腐蚀"的免疫机制。**

承诺腐蚀来自四种力：

| 腐蚀源 | 表现 |
| --- | --- |
| **时间** | 短期决策慢慢违背长期价值 |
| **人 / AI 主体差异** | 不同读者各自理解偏移 |
| **诱惑** | 局部最优 vs 全局目标的拉扯（"就这一次破例"累积成新常态） |
| **遗忘** | 上下文丢失 / 注意力分散 / 新人加入不知背景 |

**宪法不是规则集合**，是让承诺**跨时间 / 跨主体 / 跨诱惑 / 跨遗忘**仍能保持的机制。

### §1.2 问题意识为先（宪法的核心）

宪法的根本是**问题意识**——什么**痛点 / Gap / 困境**促使立宪。

- 不是"我们想做什么"（目标）
- 不是"我们准备怎么做"（意图）
- 是**为什么需要解决这个问题**

问题意识不明 → 宪法无出发点 → 所有派生规则脆弱。

### §1.3 问题驱动是奥卡姆的真判据

每条宪法条款都应回答："**它服务什么问题？**"

- 答得上来 → 必要 → 入宪
- 答不上来 → 不必要 → 剃 / 下沉到架构 / 细化层

"必要 / 不必要"看判断会主观；"服务什么问题"是客观可溯源的检查点。

### §1.4 元判据：每条规则都应能说出"防什么腐蚀 / 服务什么问题"

本 skill 自己也满足这一点（自指）：

| skill 内容 | 防什么腐蚀 / 服务什么问题 |
| --- | --- |
| 强制质询（§2.1） | 防"AI 用推测填空"腐蚀意图 |
| 不擅自修宪（§2.2） | 防"短期诱惑随意改宪" |
| 起源对话（§4.1） | 防"基于推测拟稿" |
| 减法优先（§6.2） | 防"sub-case 累积稀释精神" |
| 接受不完美（§0.3） | 防"形式完美主义反噬实质" |

答不上来的规则该剃。

### §1.5 核心原则：奥卡姆 + 清晰

#### §1.5.1 奥卡姆（适用所有层级）

**冗余必剃。** 制宪上尤其严苛——任何不服务于"防腐蚀 / 解决根本问题"的条款都该剃。

奥卡姆**不仅适用规范核心层，也适用经验实践层**：

- 规范核心冗余 → 削弱精神（条款互相约束 / 重复表述同一精神）
- 经验实践冗余 → 让 skill 难读 / 重复 / 自相矛盾

**判据**：每条规则用 §1.3 问题驱动审 + §1.4 元判据审，答不上"服务什么问题 / 防什么腐蚀" → 剃。

#### §1.5.2 清晰原则（双层）

宪法 / skill 文本必须满足两层清晰：

- **制定人清晰**：宪法条文的明确意图 **100% 与用户对齐**。在制宪时通过 §4.1 起源对话 + §4.7 用户最终确认，确保每条意图被用户确认。
- **后来者清晰**：文本**自洽**，能被新 Agent / 其他人**单独审视**。措辞 / 解释 / 正反例独立可读，**不依赖制定时的特定讨论背景**。

两层清晰对应**制定时 / 阅读时**两种场景：

| 场景 | 受众 | 实现机制 |
| --- | --- | --- |
| 制定时 | 当前用户 | 强制质询 + 起源对话 + 最终确认（§2.1 + §4.1 + §4.7）|
| 阅读时 | 新 Agent / 后来者 | 文本独立自洽（措辞 + 解释 + 例子完备）|

具体措辞 / 解释 / 例子可能**不需要当前用户全权知晓**——制定人主要负责意图对齐，后来者清晰由文本工程保证。

**任一缺失则文本失效**：
- 制定人清晰缺失 → 条款被推测污染，下游派生脆弱
- 后来者清晰缺失 → 文本失去独立性，只能在特定背景下解读

### §1.6 书写原则（清晰原则的具体应用）

清晰原则（§1.5.2）的具体应用，主要借鉴 `principle-derivation-v2.md` §四 反失败模式提炼。**几乎所有制宪 / 文档书写都应满足**：

#### §1.6.1 剥术语测试

每段把术语换成大白话，剩下的话**仍有具体内容** → 文字合格。
剥光后只剩**稀薄 / 重复 / 结构包装** → 删 / 重写。

#### §1.6.2 反 bloat

反馈来时**深化已有理解**，不在已有结构旁加新模块 / 新表 / 新约束。
"为反馈服务"的每次添加合起来变成 bloat —— 最难自我察觉的失败。

#### §1.6.3 反故作深沉

朴素的活动用朴素的语言。**不要给本来朴素的活动起学术名字**——加包装但没增加任何操作内容。

#### §1.6.4 反空话

每条都有**具体内容**（实例 / 数据 / 推理链 / 具体反例）。没具体内容的条款应**删**，不是改写成更抽象的形式。

#### §1.6.5 优先朴素语言

能用日常语言说清的**不用术语**。术语只在**没有日常等价**或**精度损失大**时使用。

**这 5 条适用本 skill 自身**——按 §10 自指验证，本 skill 应回头按这 5 条检查自己（具体协议见 §4.8 审查 + §10）。

## §2 AI 工作硬约束（强制）

这一章是 skill 唯一的硬约束。约束 AI 行为，不约束宪法内容形式。

### §2.1 强制质询

**如果起源 / 价值 / 边界不清，AI 必须向用户质询，不能基于推测拟稿。**

强制质询触发：

- 项目目标不明确（"它要解决什么问题"）
- 利益相关方不明确（"谁读 / 谁被约束"）
- "项目不做什么"没说
- 价值冲突优先级不明
- 任一条款溯不回明示意图

质询不是签到——是避免"基于推测的条款都是脆弱的"这个根本风险。

### §2.2 不擅自修宪

所有修宪必须经用户**显式确认**。AI 不能擅自修宪。

修宪不同于普通文档修订——它影响项目根本约束 / 长期承诺。AI 可以**提议**修宪议案，但**实施**必须用户同意。

### §2.3 不基于推测拟稿

如果意图未确认（起源 / 价值 / 边界有空白），AI 必须先质询而非脑补。

基于推测的条款都是脆弱的——后续都可能因为推测错而被推翻。

## §3 宪法形态（学现实宪法的实践推荐）

### §3.1 篇幅基线

现实宪法篇幅参考：

- 罗马十二铜表法 ~1000 词
- 美国宪法 ~7500 词（含 27 修正案）
- 法国人权宣言 ~700 词
- 联合国宪章 ~10000 词

**典型篇幅 50-200 行**。超此范围大概率冗余。**不是硬上限**——能解释为什么超即可。

### §3.2 正交性（理想）

宪法条款应**相互独立**：

- 条款 A 不依赖条款 B 才成立
- 条款 A 不约束条款 B
- 删除 A 不影响 B 的语义

实际很难 100% 正交，80-90% 即可。

### §3.3 派生而非穷尽

宪法不 enumerate 所有情况。提**根本性原则**，让派生层（架构 / 法律 / 司法解释）做具体应用。

### §3.4 接受不完美 + 明示冲突消解

- 性质间天然冲突时，**直陈优先级 / 消解规则**（如"上限保证 0 误报第一"）
- **不为每个具体冲突加 sub-case**
- 接受少量"原则 + 例外"如果例外真实清晰
- **不强制 disjoint union 分解**（K → Y + X 是 prompt 技巧，不是宪法精神 — 人类能处理"原则 + 例外"模式）

### §3.5 抽象层级

判据：**删了它，整套系统的精神是否发生质变？** 质变 → 入宪法，否则 → 不入。

- **入宪法**：根本问题意识 / 不可妥协的价值 / 边界声明 / 性质 / 冲突消解
- **入架构**：模块切分 / 接口契约 / 数据流
- **入细化**：工程现实判断（什么算"复杂" / "简单"）/ schema / sub-case 列举 / 流程指令

## §4 制宪过程（实践推荐）

### §4.1 起源对话（强制，先于一切拟稿）

AI 必须先与用户确认 5 个问题：

1. 这个项目的**根本问题**是什么？（一句话）—— **问题，不是意图**
2. 解决的现实**痛点 / Gap** 是什么？
3. 它**不**解决什么？（边界）
4. **谁读**这份宪法？**谁被约束**？
5. **不可妥协价值**是什么？（冲突时优先什么）

任一答案不清 → 质询，不能继续。

### §4.2 意图提炼

把起源对话的答案抽象为：

- **目标**（一句话）
- **价值**（3-5 条）
- **边界**（3-5 条）

### §4.3 性质提取

从目标 + 价值 + 边界中提取**根本性质**：

- 性质数量典型 3-7 条
- 超 7 条 → 大概率没真正抽象到位 → 重新提取

### §4.4 派生原则

每条性质派生 1-3 条派生原则。仍在宪法层（不进架构）。

### §4.5 冲突消解

枚举性质 / 派生原则间可能的冲突 → 给消解规则。**不为每个具体冲突加 sub-case**。

### §4.6 反模式扫描

按 §5 清单逐条扫，发现就退回 §4.3 重新提取。

### §4.7 用户最终确认

宪法稿出来后，AI 必须向用户**逐条确认**：

- 每条性质："这是否符合你的根本意图？"
- 每条边界："是否真的不做？"
- 冲突优先级："这个优先级是否符合你的意图？"

确认后才落稿。

### §4.8 审查协议：先反后反反（high-strength 审视精神性陈述）

宪法 / skill 落稿后或任何重大修订时，审查必须满足：

1. **独立审查者**：审查者不是制定者本人 / 不在同一 session 的 AI——独立人 / 独立 agent / 跨 session AI
2. **先反立场（挑刺）**：审查者用反立场挑刺，给具体理由 + 行号 + 引用
3. **后反反立场（counter-challenge）**：派另一独立 agent / 第三方按挑刺逐条 counter
4. **counter-challenge 方法**：**反复高强度审视精神性陈述**（宪法层 / 下级管制法律 / 派生原则）——精神已明示则挑刺不成立 / 精神没明示则挑刺成立
5. **挑刺只在 counter-challenge 站不住后才落地**

单边审查会过度严苛 / 误剃好东西。"先反后反反"是经验性必要平衡。

#### §4.8.1 为什么是"反两次"（实用主义经验理由）

本协议**不是绝对正确**，是基于**实用主义经验**作为推荐性公理：

- LLM 本质上"**只要转换立场就能挑出不少问题**"——挑刺成本低
- **挑刺**（反立场）→ 经验上常发现问题（高 recall）
- **confirmation**（确认立场）→ 经验上无效（几乎不发现问题）
- **反两次**（counter-challenge）→ 过滤无效 / 低效挑刺（高 precision）
- **反三次以上** → 边际收益递减（陷入 confirmation 模式）

理论上可以 counter-counter-counter-...-challenge 无限递归求反，但**经济上**"反两次"是 cost-effective 最优点：成本可控 + 高 precision + 高 recall。

#### §4.8.2 自指

本协议自己也需要被审视——但作为**推荐性 + 实用主义公理**，它不要求绝对正确，要求**经验有效**。如果将来发现"反三次"或"反一次"在某场景下经验上更优，可重新校准。

#### §4.8.3 审查与修改的工作流（引用 `/workflow` skill）

任何审查产出 + 自修改必须遵循 `/workflow` skill：

- 结果按 **决策点 vs 非决策点** 区分**明示**
- **决策点**：影响其他设计 / 需用户拍板 / 不可独立 fix
- **非决策点**：局部 fix / 文字小修 / 可 agent 批量做

**可以没有决策点**（如改动全是非决策性的），但**必须明示这种区分**——否则审查产出无法直接转化为执行。

详细工作流参考 [`workflow`](workflow.md) skill。

## §5 反模式清单（出现就该退回）

- **修修补补 / sub-case 累积**：抽象失败的信号
- **工程现实绑死**：什么算"复杂 / 简单"不入宪
- **enumerate 数字**：数字会变，性质不绑数字
- **流程指令**：流程在 detailed-design / workflow
- **派生再细化**：派生应一刀切下层
- **分析框架入宪**：理论 / 经验总结（如"两层张力" / "5 层精细化"）不入宪——理论可质疑、可补充、可被替代，宪法不可违反；混淆层级
- **意图未确认就拟稿**：基于推测的条款都是脆弱的

**判据**：每条反模式都要看程度——sub-case 累积两条 OK，二十条就有问题；不是 binary。

## §6 修宪流程

### §6.1 加法修订

必须先证明"**为什么不能在架构 / 细化里**"。能下层解决就不入宪。

### §6.2 减法修订（剃法 — 优先）

按奥卡姆，发现以下任一即可剃：

- sub-case 累积 / 工程现实绑死 / 数字 enumerate
- 派生再细化（应一刀切下层）
- 分析框架混入（理论 / 经验总结）
- 重复表述同一精神的不同条款

**减法优先**——任何时候删一条都比加一条更接近精神。

### §6.3 重构修订

性质交叉互相约束 → 提取共因后重表述。

### §6.4 修订门槛

所有修宪必须经用户**显式确认**。AI 不能擅自修宪（重复 §2.2）。

修订成本本身是免疫机制的一部分——门槛不能太低。

## §7 项目宪法特有的考虑

行动性 / 项目级宪法相对国家宪法的差异：项目有明确目标、行动落地快、有明确范围。需特别考虑：

### §7.1 测试 / 验证机制

项目宪法应明示"**怎么知道我们没偏离根本意图**"：

- 测试覆盖 / 验收准则（针对核心模块）
- 上限保证 / 下限诚实之类的检验性原则
- 反作弊 / 反妥协机制

这些与项目执行强相关，国家宪法没这层。

### §7.2 迭代与版本

项目演进比国家立宪快得多。需考虑：

- v1 / v2 / vN 的范围划界（哪些下版本再说）
- 已知 v1 砍项的诚实声明
- 时效性承认（"本宪法是 v1 时点的快照"）

### §7.3 部署与运行环境约束

某些项目宪法需要包含：

- 平台范围（macOS / Linux / cross-platform）
- 部署形态（库 / 服务 / 端到端工具）
- 运行环境前提

但**注意**：环境约束容易腐蚀成"工程现实绑死"。能在细化层说就别入宪。判据：删了它项目还是项目吗？

### §7.4 模块优先级

多模块项目应明示：

- 哪些是 first-class（长期承诺）
- 哪些是 non-first-class（应用展示 / 时效性强）

避免次要模块的优先级抢核心模块。

## §8 现实宪法学习案例

**优秀实践**：

- **美国宪法第一修正案**：一条性质（不得制定法律剥夺言论自由）+ 多个具体保护——展示"性质 + 派生"
- **联合国宪章序言**：意图明示（"决心避免人类陷入两次世界大战之苦"）——展示问题意识为先
- **法国人权宣言**：边界声明（"自由 = 不损害他人的一切行为"）——展示用边界定义性质

**避免反例**：

- **苏联式宪法**：把工程现实判断写进去（细节绑死）
- **任何超 500 行的"宪法"**：大概率有冗余 / 派生再细化
- **把"两层张力 → PA/PB/PC..."这种分析框架入宪**：理论可质疑，但宪法不可违反，混淆层级会限制后续思考

## §11 规范核心 + 经验实践（二元结构）

### §11.1 结构

本 skill 自己满足"**规范核心（normative core） + 经验实践（empirical practice）**"二元结构：

| 层 | 章节 | 性质 |
| --- | --- | --- |
| **规范核心** | §0 + §1 + §2 | 元说明 / 元问题意识 / 硬约束 — **精神 + 派生原则**，相对稳定 |
| **经验实践** | §3 - §10 | 现实宪法学习 / 制宪过程 / 反模式 / 修宪流程 / 项目宪法特有 / 案例 — **经验性、可演化、可替换** |

### §11.2 合法性约束

任何"经验实践"必须同时满足：

- **正向贡献**：服务于规范核心定义的问题（防腐蚀 / 问题驱动 / 意图对齐 / 清晰）
- **反向兼容**：不与规范核心冲突

满足两条的实践形态**理论上开放**（不止一种实践方案合法），但实践上由经验积累形成**已知有效模式**（如"派生而非穷尽" / "接受不完美" / "强制质询"）。

### §11.3 演化与取舍

- 规范核心相对稳定，经验实践会演化 / 被替换
- 不同实践方案之间可能冲突 → 取舍（trade-off）
- skill v1 → vN 时**核心应保稳，§3+ 经验性章节可大改**

### §11.4 与"宪法 vs 法律"的对应

类比现实法律体系：

| 项 | 现实宪法 | 现实法律 |
| --- | --- | --- |
| 抽象层 | 精神 + 不可让渡 | 具体规则 + 操作流程 |
| 修订门槛 | 极高（修宪程序） | 中等（立法程序） |
| 数量 | 少（一国一宪） | 多（成百上千） |
| 演化速度 | 慢 | 快 |

本 skill 的 §0-§2（规范核心）≈ 宪法层；§3-§10（经验实践）≈ 法律 / 法规层。**两层都受奥卡姆约束**（§1.5.1）；**两层都受清晰原则约束**（§1.5.2）；**两层都受审查协议约束**（§4.8）。
