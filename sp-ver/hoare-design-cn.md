# HoareDesign — 从代码反向推导设计规约

在目标代码没有显式设计文档时，通过系统性地检查每个函数的**所有调用方**（call sites），反向推导出一份**描述性设计规约**。产出反映代码**实际在做什么**，而不是它"应该"做什么。

## 使用时机

- 目标代码**缺乏设计文档** — `/hoare-audit` 没有规约无法继续。
- 现有的设计文档可能已经**过时**，需要一份新的基准。
- `/hoare-prompt` 需要一份规约来锚定其 Requires/Ensures 推导。

## 核心原则

**当前代码即预言机。必须从所有调用方推导 Pre/Post 条件，严禁依赖直觉。**

一个函数如果没有校验 X，不能直接说是"漏了校验" — 如果所有的调用方都校验了 X，那么"调用方保证 X"就是合法的前置条件。

## 执行流程

### Phase 1 — 单函数：从实现提取规约

只读函数本身的实现，提取：

```
函数: <模块>::<函数名>
签名: <实际签名>

内部守卫 (代码所做的验证/断言/提前返回):
  - <逐条列出>

返回行为:
  - 成功: <返回什么 / 变更了什么状态>
  - 错误: <每种错误变体及其触发条件>
  - Panic: <每个 expect!/unwrap! 及其编码的假设>

副作用: <I/O、产生子任务、状态突变、日志 — 或"无 / 纯函数">
```

**禁止在此处写"缺少校验"。只描述它客观做了什么。**

### Phase 2 — 单函数：检查所有调用点推导 Pre/Post

对每个函数，**全局搜索所有调用点**并追踪：

```
调用点: <调用方模块>::<调用方函数>, 第 N 行

调用方在调用前建立的状态:
  - <调用方在此调用前做的校验/状态设置>

调用方如何使用返回值:
  - <返回值怎么被消费 — 调用方需要返回值的什么性质为真>

调用方的错误处理:
  - <调用方处理了 Err？传播了？忽略了？>
```

**聚合所有调用点**来推导：

```
前置条件 (所有调用方保证的交集):
  - <只列出每一个调用方都建立的条件 — 最弱的公共保证>

后置条件 (任一调用方依赖的并集):
  - <列出任何调用方从返回值中依赖的全部性质>

调用方契约 (由调用方而非函数自身强制的前置条件):
  - <由函数的 unwrap/expect 推导 — 若调用方未保证则会 panic>
  - <由缺失校验推导 — 函数不检查 X 但没有调用方因此出错>
```

### Phase 3 — 模块级：推导责任边界

```
模块: <模块名>

职责: <此模块拥有什么>
不变式: <跨所有公开方法调用保持的性质>
副作用范围: <此模块做/不做什么 I/O>
责任分配: <谁检查什么 — "调用方为被调用方 Y 保证 X">
错误哲学: <快速失败 vs 宽容失败 vs 阅后即焚 — 从实际模式推导>
```

### Phase 4 — 识别"故意忽略"的设计

扫描揭示**有意为之的设计决策**的模式：

- 函数可以校验 X 但没有，且所有调用方都校验了 X → "通过设计将责任分配给调用方"
- `let _ =` 或 `.ok()` 处理 Result → 有意的错误抑制（阅后即焚）
- 函数返回 `T`（不可失败）尽管子操作可能失败 → 调用方保证了成功条件
- 函数的契约比它"可以有的"更简单 → 有意的简化 trade-off

记录为："不做 X — 所有调用方代替做 X → 责任分配给调用方"

### Phase 5 — 意图分类（非决策性 vs 决策性）

这是对接审计循环（Audit Loop）的关键步骤。意图推导必须优先保证自动化推进。

- **非决策性 (高度一致的意图)**：所有调用方行为完美一致（例如：每个调用方都在传参前校验了 ID）。意图明确无歧义。**自动采纳**该推导规约为基准。
- **决策性 (冲突的意图)**：调用方互相矛盾（例如：调用方 A 判空了，调用方 B 直接传裸数据），或调用方的行为与函数内部假设发生明确冲突。这是架构设计上的冲突。
  - **行动**：标记为"决策性"。**严禁阻塞当前流程。**
  - **审计行为**：在当前轮次中，临时采用最弱的公约数作为契约以防误报；将冲突记录为"决策性冲突"，在审计循环的人类决策门控中呈报。

## 产出格式

写入 `docs/design/hoare-derived-spec.md`（或由调用者指定的路径）：

1. **模块概述** — 每个模块一段话的总结
2. **函数规约** — Phase 1 + Phase 2 合并：实现行为 + 调用方推导的 Pre/Post
3. **责任链条** — 谁向谁保证什么（附调用点证据）
4. **模块契约** — Phase 3 聚合
5. **有意忽略清单** — Phase 4 "通过设计不做"的列表
6. **意图分类日志** — 自动采纳的"非决策性"规约，及留待人类处理的"决策性冲突"
7. **未验证的假设** — 没有任何调用方建立的前置条件 (⚠ 标记) — 真正 bug 的候选

## 规则

- **描述性，非规范性。** "代码做了 X"，不是"代码应该做 X"。
- **所有调用点，不是有代表性的调用点。** 从 5 个调用方中的 3 个推导的前置条件是错误的 — 你需要全部 5 个。
- **缺失是关于责任的信息，不是关于 bug 的信息。** 缺少的校验意味着责任在别处，除非检查完所有调用方后证明并非如此。
- **当前代码在被证明有误前视为正确。** 此规约假设代码是对的。`/hoare-audit` 才是你质疑它的地方。
- **引用代码，不要复述。** 为每个前置条件来源引用确切的行。复述会引入语义漂移。
