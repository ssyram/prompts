# HoarePrompt 审计循环 — 持续的正确性验证

使用 HoarePrompt 方法论执行持续的代码正确性审计（方法论详见 `/hoare-prompt`）。该循环自动隔离根因、自动执行非决策性修复，并将涉及架构取舍的决策性问题累积起来交由人类决策。

## 审计目标

$ARGUMENTS

如未指定目标，审计当前工作目录。

## Step 0 — 规约门控 (强制要求)

**没有规约，绝不开始审计。** 没有规约的审计会生成听起来很自信但可能与有意设计决策相矛盾的发现。

**Step 0 未完成前，严禁开始 Step 1。**

### 0a. 检查显式设计文档

搜索设计文档：`docs/design/`、`DESIGN.md`、`docs/architecture.md`、`docs/detailed-design.md` 等。同时检查 `CLAUDE.md` 中与设计相关的段落（已知限制、设计决策、不变式）。

### 0b. 设计文档存在 → 提取契约作为基准

对每个目标函数，从设计文档中提取：
- **签名**（不可失败 vs 可失败 — 设计写 `-> Self` 而代码返回 `Result` 构成违规）
- **列出的错误**（只有这些，不多不少 — 新增的是偏离）
- **"调用方保证"** / 前置条件分配（谁负责什么）
- **"有意不做"** 的决策及其理由（不是 bug — 是 trade-off）
- **可用性/正确性权衡**（"阅后即焚"、"允许丢失"、"尽力而为"、"仅防御性"）

这些是**基准真相**。与设计一致的代码行为就是正确的，即使审计者的直觉觉得"缺了点什么"。

### 0c. 无设计文档 → 先运行 `/hoare-design`

**严禁直接进入 Step 1。** 改为：
1. 通知用户："未找到 [目标] 的设计文档。正在运行 `/hoare-design` 推导描述性规约。"
2. 运行 `/hoare-design`，应用 Phase 5 分类：
   - **非决策性规约**（调用方行为一致）：立即自动采纳。
   - **决策性冲突**（调用方行为矛盾）：标记待人类审查。不阻塞。本轮按最弱契约执行。
3. 将推导出的规约交用户审阅后再继续。

### 0d. 记录规约来源

在每轮审计输出的顶部声明：
```
规约来源: docs/design/detailed-design.md (显式设计文档)
— 或 —
规约来源: /hoare-design 输出 (从所有调用方反向推导)
— 或 —
规约来源: 用户口头描述
```

**绝不以"规约来源: 无"继续。** 如果无法获取规约，告知用户并停止。

## 执行流程

### Round N (重复直到两个队列都清空):

**Step 1 — 声明部署上下文 + 选择审计维度。**

**1a. 声明部署上下文。** 写一份部署概况，校准"什么是真实威胁"：

```
部署上下文:
  - 执行模型: <CLI 工具 / 长运行服务 / 库 / 插件 / ...>
  - 信任边界: <谁提供输入 — 用户、LLM、网络、内部模块>
  - 持久化模型: <临时 / 内存 / 磁盘 / 数据库>
  - 并发模型: <单线程 / 异步单线程 / 多线程 / 多进程>
  - 威胁模型: <仅本地用户 / 多租户 / 面向互联网 / ...>
```

此概况界定所有后续分析的范围。

**1b. 选择审计维度。** 选择 3-6 个对**此目标**真正重要的维度。不要默认使用固定列表 — 不同项目需要不同的角度。

候选维度供*参考*（非穷尽、非强制 — 按需选取、扩展或发明新的）：

- **崩溃安全 / 空值安全** — 空指针解引用、未捕获异常、动态边界的形状不匹配
- **功能正确性** — 契约、不变式、分支穷尽性、循环终止（NSP 式正向追踪）
- **外部 API 契约** — 对框架/库行为的假设，需对照源码验证
- **资源生命周期** — 定时器、句柄、订阅、文件描述符，在所有路径上释放
- **并发 / 持久化完整性** — 竞争条件、部分写入恢复、原子性、fsync 顺序
- **安全 / 输入验证** — 注入（shell、SQL、环境变量、prompt）、路径穿越、SSRF
- **错误传播与可观测性** — 静默失败、被吞没的错误、误导性的返回值
- **协议 / 状态机正确性** — 不可达状态、死锁、跨转换的不变式违反
- **性能 / DoS** — 无界循环/分配、病态输入、二次复杂度
- **前向/后向兼容** — Schema 迁移、磁盘格式、线上协议
- **隐私 / 数据处理** — PII 流向、保留期限、日志和遥测中的脱敏
- **确定性与可复现性** — 时间、随机性、迭代顺序、浮点
- **可移植性** — 平台假设（路径、Shell、换行符、编码）
- **幂等性** — 重试安全、精确一次 vs 至少一次语义
- **权限 / 授权** — 权限提升、工具暴露边界
- **升级 / 回滚** — 状态是否在重启后存活、版本化工件、迁移可逆性
- **数值 / 边界正确性** — Off-by-one、溢出、下溢、整数与浮点域
- **竞争之外的并发** — 饥饿、优先级反转、死锁环
- **连接 / 会话生命周期** — 连接/断开时机、断开副作用、PID/会话映射生存期假设、重连策略
- **可测试性 / 可观测性** — 生产环境中出了问题能否发现？

在继续之前，说明你选择的维度以及*为什么每个与此目标相关*。

**Step 2 — 为每个维度并行启动一个 Agent。** 每个 Agent 从头阅读源码，互不看对方的发现。各自写入 `docs/audit/RoundN/audit-<维度>.md`。Agent 必须以框架/库源码为真相来源，而非文档或假设。

**Step 3 — 汇总发现。** 阅读所有报告，应用过滤：

**3a. Hoare 发现 vs 代码走查级挑剔 — 证据要求。** Hoare 发现必须具备：
- 一个被违反的具体 Pre/Post/不变式，并且
- 一个触发违反的具体反例（输入、状态或执行路径），并且
- **代码类发现**：必须提供具体的测试用例、PoC 脚本或逐步复现步骤，证明 bug 确实会触发。"理论上可能发生"不够——要展示它发生。
- **文档类发现**（过时内容、前后矛盾、缺少标记）：必须三重确认文档中没有其他部分缓解了该问题（如 changelog 条目、免责声明段落、交叉引用）。如果文档其他位置存在缓解措施，降级为 nit，除非该缓解对文档的主要使用模式（如按 section 查阅 vs 顺序阅读）明显不足。

两者都缺少的是**代码走查级挑剔** — 需要时单独记录到 `docs/audit/RoundN/nits.md`。

例外：死代码 / 过时注释 / 未使用参数仍是有效发现（理由是维护性信噪比），但仍需引用确切位置并解释为什么该过时内容会造成混淆或维护负担。

**3b. 运行时影响追踪。** 对每个候选发现，追踪该问题在部署上下文下是否**在运行时可观测到**：
- 违规输入/状态是否真的到达了受影响的代码路径？
- 该路径从系统实际入口点是否可达，还是只从死调用方可达？
- 如果只在不匹配的威胁模型下可达 → 降级为**加固建议**。

**3c. 对存活问题分类:**
- VULNERABLE / DISPROVEN / REFUTED / LEAK → 修复**候选**（须通过 Step 4）
- GUARDED / PARTIAL / UNVERIFIABLE → 逐案评估
- SAFE / PROVEN / CONFIRMED / NONE → 无需行动

**Step 4 — 独立质疑（驳回优先）。** 每个候选发现必须由 ≥2 个未参与其产出的全新 Agent 独立**质疑**。质疑者的任务是**推翻**该发现——寻找它错误、夸大或基于误读的理由。只提供新视角——不给质疑者看原始审计报告，只给具体主张。

**原理**：确认偏误是审计误报的第一大来源。被要求"确认"发现的 Agent 会寻找支持它的证据；被要求"推翻"发现的 Agent 会寻找反对它的证据。经受住推翻尝试的发现远比未经质疑的发现可靠。

向每个质疑者提供：
- 具体主张（文件、函数、描述、拟议严重程度）及相关源文件。
- Step 1 的部署上下文。
- 显式指令："你的任务是找出这条发现错误的理由。寻找反驳它的证据。保持对抗性。"

要求在 `docs/audit/RoundN/challenge-<发现编号>.md` 中给出以下判定之一：
- **UPHELD**：质疑者尝试推翻但未能成功。发现成立。质疑者必须说明尝试了哪些推翻角度以及为什么失败。
- **WEAKENED**：发现部分正确但被夸大。质疑者必须说明哪些正确、哪些被夸大，附证据。
- **REJECTED — deployment context mismatch**：主张假设了不匹配的威胁模型。
- **REJECTED — not reachable**：主张假设的状态被不变式 Y 阻止；引用不变式。
- **REJECTED — misreading**：原审计者误读了代码；解释代码实际做了什么。
- **REJECTED — mitigated**：问题孤立存在但被审计者遗漏的另一机制缓解了；引用缓解措施。
- **INCONCLUSIVE**：无法决定；说明还需要什么信息。

最多 6 个质疑者并行。每条发现需要 ≥2 个独立质疑者。

**判定聚合**：
- 所有质疑者 UPHELD → 发现进入 Step 4.1
- 所有质疑者 REJECTED → 发现丢弃（记录到已拒绝发现日志）
- 判定分裂（有分歧）→ 发现带"争议"标记进入 Step 4.1
- 所有质疑者 WEAKENED → 发现以调整后的严重程度/范围进入下一步

**Step 4.1 — 反质疑（存活发现的二次检验）。** 对每条通过 Step 4 的发现（UPHELD 或争议），再启动一个全新 Agent 作为**反质疑者**。该 Agent 收到：
- 原始主张
- 质疑者判定摘要（尝试了哪些推翻角度）
- 指令："之前的质疑者试图推翻此发现但未成功。做最后一次尝试。寻找他们遗漏的角度。"

如果反质疑者也无法推翻 → 发现**确认成立**。如果反质疑者找到有效的推翻理由 → 发现标记为**争议项**，必须在 Step 8 中将双方证据呈报人类。

**Step 4.2 — 主审亲自验证。** 主审（你）必须亲自阅读源码验证每条确认发现的关键证据。不能仅依赖 Agent 报告。对每条发现，声明："我在 [file:line] 验证了 [具体证据]"或"我无法验证 [主张]——Agent 可能误读了"。

对争议项，将各方论点和证据编纂为平衡的摘要，供人类决策门控使用。

**Step 4.5 — 归约与分类 ("只诛首恶" / 隔离根因)。**

*此步骤每一轮循环都必须执行，不只是最后一轮。*

对所有已确认的发现执行归约流水线：
1. **去重**：合并从不同角度指出同一逻辑缺陷的发现。
2. **只诛首恶 (隔离根因)**：如果发现 B 纯粹是由发现 A 导致的，丢弃 B，只保留 A。只有根因存活。
3. **去已消解**：丢弃在之前轮次的修复中已被消除的发现。

将归约后剩下的发现分为两个桶：
- **非决策性 (自动化处理)**：明显的局部逻辑错误、越界、空指针、明确违反无歧义规约的 bug — 有清晰正确修复的无歧义 bug。
- **决策性 (人类介入)**：架构取舍、API 契约变更、模块边界重构、冲突的规约（来自 Step 0）、设计哲学选择。

**Step 5 — 自动修复非决策性问题。** 全自动执行所有非决策性发现的修复。不需要请示人类。应用最小化补丁，直击 Step 4.5 锁定的根因。对非 trivial 的修复，记录根因和修复理由（问题是什么、为什么这样修、修复后预期行为）。

**Step 6 — 验证扫尾。** 启动 Agent 验证 Step 5 的每个修复（每个修复点 PASS/FAIL）。写入 `docs/audit/RoundN/verify.md`。条件允许时补充回归测试。如有 FAIL，返回 Step 5。

**Step 7 — 收敛检查。**
- 如果队列中**还有任何**非决策性发现（或 Step 5 的修复引入了新问题）→ 增加轮次 Round N+1，返回 Step 1。
- 如果非决策性发现清零 → 进入 Step 8。

**Step 8 — 人类决策门控。** 向人类呈报所有累积的**决策性发现**：
- 概述每个架构取舍的利弊。
- 如适用，包含 Step 0 的规约冲突上下文。
- 等待人类决策。

**Step 9 — 执行决策并重启循环。** 执行人类拍板的决策。由于这些决策改变了系统的规约/架构，**必须重启循环**（进入 Round M，从 Step 0 开始），在新规约下重新评估整个代码库。整个大循环持续运转，直到"非决策性"和"决策性"两个队列双双清零。

### 最终输出

写入 `docs/correctness-audit.md`，包含：

1. **执行摘要** — 维度 → 结果计数表（提出 / 确认 / 已修复）
2. **已证明对象** — 严格范围：精确证明了什么，依赖什么假设
3. **跨边界契约** — 所有已验证假设的表格
4. **发现与修复** — 按轮次的表格：提出了什么、确认了什么、做了什么
5. **已拒绝/不确定的发现** — 在 Step 4 中丢弃的候选发现及原因（防止后续轮次重复提出）
6. **决策性发现日志** — 所有决策性项目、人类决策及其理由
7. **剩余局限** — 无法完全证明的内容 (PARTIAL/UNVERIFIABLE)
8. **假设注册表** — 每个外部依赖，供 API 变化时重新验证

## 规则

- **保留痕迹**：每轮的维度报告、确认判定和验证扫尾都保存在 `docs/audit/RoundN/` 下。严禁删除。
- **每轮新视角**：新 Agent 严禁阅读之前的 `docs/audit/RoundN-k/` 目录。显式指示："不要读取 `docs/audit/` 的任何子目录"。
- **框架 = 真相来源**："文档说了 X"不是证据。`module.rs:line_N` 才是证据。
- **先修后证**：如果代码可以改来消除某个"已知局限"，就不要接受它。
- **严格判定**：PROVEN 意味着每条路径都被追踪了。跳过了边界情况就只能说 PARTIAL。
- **源码中不加标注**：证明存放在 `docs/` 中，不作为行内注释。
- **独立质疑是强制要求**：没有经过 Step 4 质疑的发现不能成为修复。没有触发场景或可靠理由的猜测直接拒绝并记录。未经任何质疑者尝试推翻的发现视为未审核。
- **已拒绝的发现被记录，不被遗忘**：防止后续轮次重复提出同样的误报。
- **死代码 / 过时注释 / 未使用参数是有效发现**：不需要运行时触发器 — 理由是信噪比。
- **非决策性优先**：永远优先自动推进局部修复。只有在决策从根本上改变系统设计时才打扰人类。
- **只诛首恶**：Step 4.5 的修剪归约在每一轮都是强制要求。绝不修补症状。
- **持续规约同步**：如果某次修复改变了行为边界，必须同步更新规约。
- **熔断机制**：如果同一个模块在连续 3 轮以上都有确认发现，停止打补丁。对该模块做根因分析，找出修复为什么一直在引入新问题。
- **驳回优先，而非确认优先**：对任何发现的默认立场是怀疑。发现必须经受住主动的推翻尝试才能成为可执行项。未经质疑的发现弱于经质疑后存活的发现。
- **证据高于断言**：代码类发现，"我认为这可能发生"不是证据——展示具体的测试用例、PoC 或逐步追踪来证明问题会触发。文档类发现，展示确切的错误文本，并确认文档中没有其他部分纠正或缓解了它。
- **争议项交由人类**：如果质疑者和反质疑者意见不一，将双方证据呈报人类。不以多数票解决争议——呈现论点。
